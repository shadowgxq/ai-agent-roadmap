> **一句话目标**:跑通「发消息 → 流式收回复 → 维护多轮历史 → 精确算 token/费用」的最小闭环,并用 FastAPI 把流式输出暴露成浏览器可见的 SSE 接口。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:无(第 1 周,只需装好 Python 3.12+ 和 uv,并拿到 ANTHROPIC_API_KEY)

## 1. 核心要掌握(能口头讲清才算过)

### LLM API 是无状态的
服务端不保存任何会话记忆,每次请求都是独立的——你必须把**全部历史消息**重新发过去,模型才「记得」前文。这意味着:所谓"多轮对话"完全是客户端拼出来的假象;会话状态存在你的内存/数据库里,不在 Anthropic 那边;历史越长,每轮请求的 input tokens 越多、越贵。面试追问「那 ChatGPT 为什么记得上文」——因为它的前端替你维护并重发了历史,API 层面依然无状态。

### messages 结构与 system prompt
在 **Anthropic Messages API** 里,`messages` 是一个 `{role, content}` 数组,role 只有 `user` 和 `assistant` 交替;`system` 不放在 messages 里,而是顶层独立参数。  
在 **OpenAI-compatible** 里,`system` 通常作为 `{"role": "system", "content": ...}` 放进 `messages` 数组。  
不管走哪套,system prompt 的职责都一样:定义身份、边界和行为规范。Anthropic 的 content 可以是字符串,也可以是 content block 数组(text/image/tool_use/tool_result)——W2 的 tool use 就靠 block 数组,现在就要习惯这个结构。

### context window vs max_tokens
context window 是「输入 + 输出的**总容量**上限」(模型固有属性,如 200k);`max_tokens` 是「本次请求**最多生成多少输出**」(你传的参数)。两者关系:input_tokens + max_tokens 不能超过 context window,否则直接报错。max_tokens 设太大不会多花钱(按实际输出计费),但会压缩可用输入空间、部分场景影响排队/限流配额;设太小则回复被截断,`stop_reason` 变成 `max_tokens`——要会检查这个字段。

### token 计费模型
input 和 output **分别计价**,output 单价通常是 input 的数倍。每次响应的 `usage` 字段给出精确的 `input_tokens` / `output_tokens`。关键推论:10 轮对话中,第 10 轮的 input ≈ 前 9 轮所有内容之和,**总成本随轮数呈平方级增长**——这就是 W7 要做 auto-compact 的根本原因,现在先用打印数字亲眼看到它。

### streaming 事件流
流式响应不是「一段一段的文本」,而是一串**类型化事件**:`message_start`(含初始 usage)→ `content_block_start` → 多个 `content_block_delta`(真正的增量文本在 `delta.text` 里)→ `content_block_stop` → `message_delta`(含最终 output_tokens 和 stop_reason)→ `message_stop`。要能说清:为什么最终 token 数在 `message_delta` 里才拿得到;SDK 的 `text_stream` 是对这些事件的高层封装,底层事件流才是 W11 做多类型前端渲染的基础。

### stop_reason 初识
Anthropic 风格字段名叫 `stop_reason`;OpenAI-compatible 常见字段名叫 `finish_reason`。  
本周最先要识别的是「正常结束」和「被 `max_tokens` 截断」这两类。Anthropic 文档里常见 `end_turn` / `max_tokens`;OpenAI-compatible 里常见 `stop` / `length`。养成习惯:**每次拿到响应先看 stop_reason / finish_reason,再用内容**——很多"模型回答突然断掉"的 bug 其实是没检查这个字段。

### 模型档位与开发成本纪律
开发调试期一律用 **Haiku 档位**(便宜一个数量级),只在验收/对比效果时切 Sonnet 档位。模型 ID 和单价写进配置而非散落代码,用本机 `claude-api` skill 查最新值,不凭记忆写。从第一天就把「每次调用都知道自己花了多少钱」变成肌肉记忆——这就是 W10 成本工程的起点。

### SSE 与 FastAPI 异步
SSE(Server-Sent Events)是 HTTP 长连接上的单向推送:`Content-Type: text/event-stream`,消息格式为 `data: ...\n\n`。FastAPI 用 `StreamingResponse` 包一个 async generator 即可。大坑:用**同步** Anthropic client 会阻塞整个事件循环,所有并发请求一起卡死——必须用 `AsyncAnthropic`。这块你有前端 SSE 消费经验,重点补的是服务端生产侧。

## 4. 自测题 + 答案要点

**1. 为什么说 LLM API 是无状态的?这对「多轮对话」的实现意味着什么?**
- 服务端每次请求独立处理,不存任何会话状态
- 多轮对话 = 客户端维护 messages 数组,每轮把全部历史重发
- 推论:历史管理是你的责任;成本随历史累积增长;漏 append assistant 回复就"失忆"

**2. context window 和 max_tokens 的区别是什么?如果 max_tokens 设太大会怎样?**
- context window = 输入+输出总容量(模型固有);max_tokens = 本次输出上限(请求参数)
- input + max_tokens > context window 时请求直接报错
- 计费按实际输出,设大不直接多花钱,但挤占输入空间;设太小则截断,stop_reason=max_tokens

**3. 一段 10 轮的对话,第 10 轮请求的 input token 大概是第 1 轮的多少倍?为什么?**
- 若每轮问答长度相近,第 10 轮 input ≈ 第 1 轮的 10–20 倍(system + 前 9 轮全部问答 + 本轮问题)
- 因为无状态,历史全量重发;总成本对轮数是平方级
- 用 Session 5 的实验表格给出你自己的真实数字

**4.(加练)stop_reason 有哪几种?拿到响应后为什么要先看它?**
- 本周会遇到 end_turn / max_tokens,W2 增加 tool_use;stop_sequence 备查
- max_tokens 意味着内容不完整,直接使用会产出半截 JSON/半截代码这类隐蔽 bug
- 正确姿势:先 switch stop_reason 再消费内容,这个习惯直接沿用到 W3 agent loop 的循环条件

## 6. 常见坑速查

- **SSE 不流式,一次性蹦出来** ｜怎么发现:浏览器/curl 等几秒后内容整块出现 ｜怎么解决:确认 `media_type="text/event-stream"`、每条消息以 `\n\n` 结尾、中间没有 buffering 的代理;用 `curl -N` 先排除前端问题
- **多轮对话"失忆"** ｜怎么发现:第二轮问"我刚才说了什么"答不上 ｜怎么解决:检查是否把 assistant 回复 append 回了 messages——这是最高频的一个 bug
- **同步 SDK 阻塞 FastAPI 事件循环** ｜怎么发现:两个终端并发 curl,第二个要等第一个完全结束才开始 ｜怎么解决:换 `AsyncAnthropic`;若必须用同步库,用 `run_in_threadpool` 包
- **流式场景拿不到 output_tokens** ｜怎么发现:usage 打印为 0 或缺失 ｜怎么解决:input 在 `message_start` 里,最终 output 在 `message_delta` 里,不要只在 message_start 取
- **API key 硬编码进代码并 commit** ｜怎么发现:git diff 里出现 sk- 开头字符串 ｜怎么解决:环境变量 + `.gitignore` 掉 `.env`;已泄露立即在控制台吊销重发
- **max_tokens 截断没被发现,拿半截内容继续用** ｜怎么发现:回复末尾突兀中断;检查 stop_reason == "max_tokens" ｜怎么解决:统一在响应处理层断言 stop_reason,截断时告警或加大 max_tokens 重试
- **凭记忆写模型单价,费用统计错一个数量级** ｜怎么发现:和控制台账单对不上 ｜怎么解决:单价只从 `claude-api` skill / 官方定价页抄,写进 pricing.py 一处维护

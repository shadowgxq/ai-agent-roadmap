# W2｜Tool Use + 结构化输出

> **一句话目标**:不看文档能手画并亲手实现完整的 tool use 回路,拥有一个「加工具只需写函数 + 装饰器」的通用工具注册表,并做出结构化输出校验重试和单轮多工具并发。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W1(异步 client、streaming、messages 维护)

> ⚠️ **硬规则**:tool use 回路是主文档规定必须**亲手写**的四个核心机制之一。本周所有回路代码(消息拼装、tool_result 回填、循环控制)禁止让 AI 代写,AI 只用来答疑和 review。这个回路就是 W3 agent loop 的前身,现在手写一遍,下周才能真正理解 agent。

## 1. 核心要掌握(能口头讲清才算过)

### Tool use 完整回路
时序:① 请求带 `tools` 参数(工具的 JSON Schema 定义)→ ② 模型返回 `stop_reason: "tool_use"`,content 里含一个或多个 `tool_use` block(name + input + id)→ ③ **你**在本地执行对应函数 → ④ 把结果包成 `tool_result` block(必须带对应的 `tool_use_id`)、以 **user** 角色 append 回 messages → ⑤ 再次调用模型,它基于结果继续,直到 `stop_reason: "end_turn"`。核心认知:**模型自己不执行任何东西,它只是输出一个"请求你执行"的结构化意图**;整个回路的循环控制、执行、回填全是你的代码在做——这就是为什么说控制权始终在工程师手里。

### tool schema 与 description 的权重
工具用 JSON Schema 描述(name / description / input_schema),Pydantic 的 `model_json_schema()` 可以自动生成 input_schema。**description 比参数名重要得多**:模型靠 description 判断"什么时候该调这个工具、参数怎么填",含糊的描述直接导致乱调或不调。好的 description 要写清:干什么、什么时候用、参数含义与格式、返回什么、何时**不该**用。这条在 W8 写 MCP server 时会再次成为成败关键。

### tool_use_id 配对规则
一次 assistant 回复里的**每个** `tool_use` block,下一条 user 消息里都必须有一个 `tool_use_id` 与之对应的 `tool_result`,一个都不能少、顺序上 tool_result 要在该条消息的最前面——否则下一轮请求直接报 400。工具执行失败也不能跳过:把异常信息作为 `tool_result` 内容并设 `is_error: true` 回传,模型看到错误后会自我纠正(换参数重试、换工具)。「错误也是信息」是 agent 工程的基本功。

### stop_reason 语义
`end_turn`(说完了)/ `tool_use`(请求执行工具)/ `max_tokens`(被截断,危险信号)/ `stop_sequence`。回路的循环条件就是 `while stop_reason == "tool_use"`。要能答:如果 `max_tokens` 截断发生在 tool_use block 中间会怎样(拿到不完整的 input,要检测并处理)。

### 结构化输出 + 校验失败重试
让模型输出符合 Pydantic schema 的 JSON 的实用手法:提供一个"提交结果"工具并引导模型必须通过它输出(tool input 天然是结构化 JSON),或直接要求纯 JSON 输出再 `model_validate_json` 解析。校验失败时**不要自己修 JSON**,把 `ValidationError` 的文本原样发回模型让它自我纠正,设重试上限(3 次)+ 兜底(抛出明确异常)。这个「用模型自己纠错」的模式贯穿后面所有周。

### tool_choice:谁决定调不调工具
默认 `tool_choice: {"type": "auto"}`——模型自己决定调不调;`any` 强制它必须调某个工具;`{"type": "tool", "name": ...}` 强制调指定工具。结构化输出场景下,「强制调用一个 submit_result 工具」是让输出 100% 走 schema 的常用手法。要能答:auto 模式下模型不调工具直接口答怎么办——要么接受(说明确实不需要),要么改 description/prompt,要么上 tool_choice 强制。

### 单轮多工具与并发执行
模型可以在一条回复里返回多个 `tool_use` block(如「北京和上海哪个热」→ 两个天气查询)。逐个 await 是串行浪费,用 `asyncio.gather` 并发执行,再把多个 tool_result 按 id 装进**同一条** user 消息回传。注意 gather 要配 `return_exceptions=True` 或逐 task 包 try/except,一个工具失败不能拖垮整批。

## 4. 自测题 + 答案要点

**1. 不看文档,能手画 tool use 的完整消息流时序图吗?**
- 四段:user 提问(请求带 tools)→ assistant 返回 tool_use block(stop_reason=tool_use)→ user 消息携带 tool_result(带 tool_use_id)→ assistant 给最终文本(stop_reason=end_turn)
- 关键点:tool_result 以 user 角色发送;每个 tool_use 必须有配对 result;循环可能重复多轮
- 用 Session 1 你亲手画的那张图自查

**2. 工具描述(description)为什么比参数名更重要?**
- 模型决策「调不调、怎么调」主要依据 description 的语义,参数名只是辅助信号
- 好 description = 用途 + 何时用/不用 + 参数格式 + 返回内容;含糊描述导致漏调/乱调
- 有 S6 的 A/B 数字支撑更佳;W8 MCP server 的成败同理

**3. 如果模型返回的 JSON 不符合 schema,最优雅的处理方式是什么?**
- 不要在代码里"猜着修"JSON;把 ValidationError 文本回传给模型自我纠正
- 必须设重试上限(3 次)+ 超限后的明确兜底(异常/默认值),防无限循环烧钱
- 更稳的形态:用 tool 的 input_schema 约束输出,天然结构化

**4.(加练)tool_choice 的三种模式分别什么时候用?**
- auto:常规对话/agent 场景,让模型自主判断
- any / 指定 tool:结构化输出、必须走某流程的场景,消除"模型口答绕过工具"的不确定性
- 用 S5 两种形态的对比数据支撑你的回答

## 6. 常见坑速查

- **漏了 tool_use_id 配对的 tool_result** ｜怎么发现:下一轮请求直接 400,错误信息提到 unresolved tool_use ｜怎么解决:回填时按 assistant 消息里的 tool_use 列表逐个生成 result,数量必须一致;写个断言检查
- **工具异常直接把服务打崩(500)** ｜怎么发现:喂个坏参数,进程栈溢出/接口 500 ｜怎么解决:registry.execute 统一 try/except,异常文本包成 `is_error: true` 的 tool_result 回传
- **结构化输出无限重试** ｜怎么发现:一次调用迟迟不返回,token 费用飙升 ｜怎么解决:硬编码 max_retries=3,超限抛异常;每次重试打日志
- **tool_result 放错位置或角色** ｜怎么发现:400 报错或模型行为诡异 ｜怎么解决:tool_result 必须在 **user** 角色消息里,且位于该消息 content 的最前部
- **gather 里一个工具挂了拖垮整批** ｜怎么发现:双工具任务里一个失败,另一个成功的结果也丢了 ｜怎么解决:`return_exceptions=True` 或每个 task 内部自兜异常,保证每个 tool_use 都有 result
- **description 含糊导致模型不调工具/瞎编参数** ｜怎么发现:该调不调、直接口算,或参数格式错 ｜怎么解决:按 S6 要点重写 description,写明何时用/参数格式/返回什么
- **流式 + tool use 混用时 input 拼装不完整** ｜怎么发现:流式模式下 tool_use 的 input 是空 dict 或半截 JSON ｜怎么解决:tool input 通过 `input_json_delta` 事件分片到达,要累积到 content_block_stop 才完整;本周可先用非流式做回路,W11 再处理流式合流
- **把时间浪费在注册真实第三方 API 上** ｜怎么发现:半个 session 过去还在折腾天气 API 的鉴权 ｜怎么解决:主文档明确说示例工具用 mock 函数返回假数据即可,本周学的是回路机制不是 API 集成
- **max_turns 忘设,模型工具调用连环套** ｜怎么发现:一次任务十几次 API 调用停不下来,费用异常 ｜怎么解决:loop 必须带 max_turns 参数(本周默认 10),超限抛异常——W3 的 agent loop 同款兜底

## 1. 核心要掌握(能口头讲清才算过)

### Agent = LLM + 工具 + 循环
agent 的本质就一句话:while 循环里反复调模型,模型说要用工具就执行并回填,说完成了就停。和 W2 的 tool loop 代码几乎同构,**质变在于控制流交给了模型**:哪一步读哪个文件、要不要跑测试、什么时候宣布完成,全由模型在循环里动态决定,你的代码只提供"手脚"(工具)和"边界"(max_turns、超时)。对照《Building Effective Agents》:步骤写死在代码里是 workflow,交给模型是 agent——这一句区分是 W5 的主题,也是高频面试题。

### 停机条件
三种终止:① `stop_reason != "tool_use"`(模型认为任务完成,正常出口);② `max_turns` 耗尽(防无限循环的硬兜底,要抛出明确异常而不是静默返回);③ 异常终止(费用熔断、用户中断——W4 完善)。要能答:为什么必须有 max_turns?因为模型可能陷入"反复读同一文件""改了又改"的死循环,没有兜底就是无限烧钱。

### system prompt 即「员工手册」
agent 的 system prompt 不是一句"你是编程助手",而是一份工作规范:身份与目标、有哪些工具与各自适用场景、**工作流程约束**(先探索理解再动手改;改完必须跑测试验证;测试不过要继续修而不是宣布完成)、输出要求。经验规律:agent 行为异常时,第一优先级排查手段是改 prompt 而不是改代码——模型"不验证就宣布完成"这类问题 90% 靠 prompt 约束解决。

### 工具设计:edit_file vs write_file
`edit_file`(old_string → new_string 局部替换)通常优于 `write_file`(整文件覆写):① 模型重写整文件时容易丢掉没注意到的部分(截断、漏 import);② 输出 token 更少更省钱;③ diff 语义清晰可审计。代价是要处理"查找串不唯一/不存在"的失败情形——报明确错误让模型自己想办法(加更多上下文行),而不是替它猜。这题主文档自测题直接考。

### 输出截断与 context 保护
`run_shell` 一次 `pytest -v` 或 `cat` 大文件可能输出几万字符,原样回填会迅速撑爆 context 且淹没关键信息。策略:每个工具的返回都设上限(如 10000 字符),超限保留头尾、中间标注 `...[truncated N chars]...`。read_file 同理要支持 offset/limit。这是上下文工程(W7)的第一课。

### 可观测性从第一天做
CLI 实时打印每一步:第几轮、模型说了什么、调了什么工具、参数摘要、结果摘要、本轮 token 数。没有这层,agent 出问题时你只能瞪着最终输出瞎猜。这既是 debug 手段,也是 W6 Web 查看器和 W9 Langfuse 接入的数据基础——现在就把"每步都是一个结构化事件"的意识建立起来。

### Session 2(~2.5h):亲手写核心循环
- 写 `src/agent/context.py`:`Context` 类管理 messages(append user/assistant/tool_results 三个方法 + 结构断言:每个 tool_use 都有配对 result)
- 写 `src/agent/loop.py`,按主文档骨架展开成真实代码:
  ```python
  async def run(task: str, max_turns: int = 30):
      messages = [{"role": "user", "content": task}]
      for turn in range(max_turns):
          resp = await llm(messages, tools=registry.schemas())
          messages.append(assistant_message(resp))
          if resp.stop_reason != "tool_use":
              return resp.text          # 模型认为任务完成
          results = await execute_tools(resp.tool_calls)
          messages.append(tool_results(results))
      raise MaxTurnsExceeded
  ```
  补全:`llm()` 封装(system prompt + tools 参数)、`execute_tools`(复用 W2 的并发 gather + 异常兜底)、`MaxTurnsExceeded` 异常、每轮 usage 累计

## 4. 自测题 + 答案要点

**1. agent loop 的终止条件有哪几种?分别对应什么情况?**
- 正常:stop_reason != "tool_use"(模型宣布完成/直接回答)
- 兜底:max_turns 耗尽 → 抛 MaxTurnsExceeded,防死循环烧钱
- 异常:费用熔断、超时、用户 Ctrl+C(W4/W6 完善);另需警惕 stop_reason=max_tokens 的截断情形

**2. 为什么 edit_file(局部替换)通常比 write_file(整文件覆写)更好?**
- 整文件覆写时模型易丢内容(截断/漏掉没读到的部分),局部替换只动目标行
- 输出 token 少 → 便宜、快;改动即 diff,可审计
- 代价:要处理查找串不唯一/不存在,报明确错误让模型自己补上下文重试

**3. system prompt 里为什么要显式要求「改完跑测试」?**
- 模型默认倾向"改完即宣布完成",不会自发验证——这是 W3 常见坑之一
- 显式流程约束(改完必须跑测试、不过继续修)能把成功率显著拉高,且是最便宜的优化手段
- 这就是"用确定性工程手段约束不可靠模型"的具体体现;W4 用 eval 数字可以证明这条约束的价值

## 6. 常见坑速查

- **模型反复读同一个文件死循环** ｜怎么发现:CLI 实时日志里连续多轮同样的 read_file 调用,轮数只增不减 ｜怎么解决:system prompt 加工作流程约束,或记录已读文件、重复读时在 tool_result 里提示(S6)
- **run_shell 输出 10 万字符撑爆 context** ｜怎么发现:某轮之后 input tokens 暴涨,甚至请求报超限错误 ｜怎么解决:工具层统一输出截断——保留头尾、中间省略并标注截断字数(S4)
- **模型改完不验证就宣布完成** ｜怎么发现:agent 说"已修复"但你手动跑 pytest 仍红 ｜怎么解决:system prompt 强制「必须运行测试确认,测试不过不许宣布完成」(S4/S5)
- **edit_file 查找串在文件里不唯一** ｜怎么发现:替换发生在错误位置,或多处被改 ｜怎么解决:不唯一时直接报错并提示"提供更多上下文行",让模型自己重试(S3 单测覆盖)
- **agent 直接改坏了 fixture 原件** ｜怎么发现:第二次跑同一 case 结果诡异,git diff 显示 fixture 被改 ｜怎么解决:每次复制到独立临时目录再跑——这也是 W4 eval 的必备纪律
- **shell 命令挂死(如交互式命令等输入)** ｜怎么发现:某轮卡住不动,无超时无输出 ｜怎么解决:wait_for 超时 + 杀进程,把 timeout 信息作为 tool_result 回传让模型换命令

# W4｜Evals 起步 + 可靠性(评估驱动开发)

> **一句话目标**:本周结束时,你能一条命令跑完一批固定 eval case 得到通过率报告,拿到基线数字,之后每次 prompt/工具改动都用数字验证;agent 具备超时/重试/费用熔断护栏,并且已开启 prompt caching。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W3(agent-mini CLI 能在真实小仓库修 bug 并自己跑测试)

## 1. 核心要掌握(能口头讲清才算过)

### 为什么 evals 先行
- agent 的改动(改 system prompt、改工具 description、换报错措辞)对结果的影响是**非线性且反直觉的**,靠"感觉变聪明了"判断必然自欺。
- 唯一可靠的方式:固定一批用例,每次改动重跑,看通过率变化——可比较、可复现、可归因。
- 追问:为什么不能只跑一遍?——agent 有采样随机性,单次结果是噪声,基线要跑 2–3 遍取区间。
- 追问:为什么第 4 周就建而不是最后?——没有 evals 的后 8 周"优化"全是玄学;有 evals,每一步都是博客素材和简历数字。

### eval case 的三要素
- fixture(带 bug 的小型测试仓库)+ task(给 agent 的任务描述)+ 验证方式(**客观命令**,如 `pytest` 退出码)。
- 关键在第三点:验证必须独立于 agent 的自述——agent 说"我修好了"不算数,`pytest` 退出码为 0 才算数。
- 好 case 还要有**区分度**:全过或全挂的 case 提供不了信息,难度要卡在"当前 agent 时过时不过"的边界附近。
- 追问:什么任务没法用客观命令验证?——"写文档""解释代码"这类,要用 LLM-as-judge(W9 才做,本周只收录能客观验证的 case)。

### 基线(baseline)与单变量归因
- 先测出"什么都不优化"时的通过率作为基线,后续所有改进都相对基线衡量。
- 每次实验**只改一个变量**(只改 prompt 的一句话、只改一个工具的 description),否则通过率变了也不知道是谁的功劳。
- 这是实验科学的基本素养,也是 W9 优化冲刺和 W10 博客的方法论地基。

### offline eval vs online eval
- offline = 固定用例集,开发期反复跑:便宜、可复现、防回归,但覆盖有限。
- online = 线上真实流量上评估(采样打分、用户反馈):覆盖真实分布,但有延迟且贵。
- 本周做的全部是 offline;两者的关系要能讲清:online 发现 offline 集里没有的失败模式,再回填成新的 offline case,形成闭环。

### 可靠性四件套
- **工具超时**:`run_shell` 遇到死循环脚本会永远挂起,必须 `asyncio.wait_for` 包住;超时后把 `"timed out after Ns"` 作为 tool_result 回传(模型能换路径自救),而不是抛异常炸掉循环。
- **LLM 调用重试**:429 限流/5xx/网络超时用指数退避(1s→2s→4s + 随机抖动)重试,上限 3 次;退避解决的是"立即重试形成重试风暴、加剧限流"的问题。
- **费用熔断**:每轮累加 input/output token 费用,超单任务上限直接抛异常终止——防止死循环烧钱,这是真金白银的护栏。
- **错误兜底**:eval runner 里单个 case 崩溃(而非 fail)不能中断整批运行,记为 error 继续跑下一个。

### prompt caching(从本周开启)
- 把请求前缀(system prompt + tools 定义)标记 `cache_control: {"type": "ephemeral"}`,后续请求前缀完全一致即命中缓存。
- 价格模型:缓存写入约 1.25 倍,命中读取约 0.1 倍,TTL 默认 5 分钟——eval 这种"短时间大量重复前缀请求"是收益最大的场景。
- 命中条件是**前缀逐字节一致**:tools 顺序、system prompt 里任何动态内容(时间戳、随机 id)都会导致永不命中。
- 有最小可缓存长度门槛(千 token 级,各模型不同,以文档为准);验证靠 usage 里的 `cache_creation_input_tokens` / `cache_read_input_tokens`。

### 实验记录(evals/LOG.md)
- 每次实验一条记录:日期、改动内容、通过率前后、成本/轮数变化、结论(保留还是回滚)。
- 变差的实验也要记——负结果同样是 W10 博客的干货("我试过 X,数据证明没用")。
- 格式见 Session 3 的示例,之后每周沿用。

## 3. 落地执行(按 session 拆分)

> 硬规则提醒:eval runner、可靠性护栏是核心机制,**必须亲手写**(AI 只答疑);fixture 仓库是体力活,**放开让 Claude Code 批量生成**,你只审核和调难度。

### Session 1(~2h):手写 eval runner 骨架
- 建目录:`evals/cases/`、`evals/run.py`、`evals/LOG.md`(空文件先占位)。
- 用 Pydantic 定义 `case.yaml` 的 schema:`task`(任务描述)、`verify_cmd`(如 `pytest -x`)、`timeout_s`、`max_cost_usd`。
- `run.py` 主流程,每个 case 依次:
  1. 把 `cases/<name>/repo/` 复制到 `tempfile.mkdtemp()` 独立临时目录(防污染);
  2. 在临时目录里调用 agent loop 跑 `task`;
  3. `subprocess.run(verify_cmd)` 看退出码判 pass/fail;
  4. 记录轮数、耗时、费用;全部跑完输出汇总表(总通过率 + 每 case 明细)。
- 先手造 1 个最简 case(可直接拿 W3 验收用过的小仓库)打通全链路。
- ✅ 完成标志:`uv run python evals/run.py` 对这 1 个 case 输出 pass/fail、轮数、费用,且 agent 是在临时目录里跑的(原始 fixture 未被修改)。

### Session 2(~2h):Claude Code 批量生成 5 个 P0 fixture
- 给 Claude Code 下指令批量造 case,prompt 要点:
  - Python 项目、文件数 ≤6、`pytest` 可直接跑;
  - 每个含 1 个明确 bug + 对应失败测试,并附 `case.yaml`;
  - bug 类型错开:off-by-one、边界条件、异常未处理、字符串/日期处理、"新增一个小函数使测试通过"。
- 你的工作是**审核**,每个 fixture 过两道关:
  - (a)修复前 `pytest` 必须 fail(红);
  - (b)自己手工打上参考修复后必须 pass(绿)——确认 bug 真实可修、测试没写反。
- 难度调节:太简单就让 AI 加干扰文件/干扰函数,太难就在 task 里加一句提示。
- ✅ 完成标志:5 个 case 全部通过"红→绿"人工验证,`git commit` 入库,commit message 注明 fixture 由 AI 生成、人工审核。

### Session 3(~1.5h):跑基线 + 开启 prompt caching
- 全量跑 `run.py` **3 遍**,记录通过率区间(比如 2/5、3/5、3/5)和平均费用——这就是基线。
- 在 LLM 调用层加 caching:system prompt block 和 tools 定义打上 `cache_control`;跑一遍 eval,确认第二次请求起 usage 里 `cache_read_input_tokens > 0`,对比开启前后单 case 平均费用。
- 在 `evals/LOG.md` 写下第一条记录,**格式示例**(之后所有记录照此格式):

  ```markdown
  ## 2026-07-24｜基线 + 开启 prompt caching
  - 改动:无 prompt 改动;system prompt + tools 定义加 cache_control
  - 通过率:基线 3/5(三遍:2/5、3/5、3/5,取中位)
  - 成本:单 case 平均 $0.31 → $0.19(cache_read 占比约 60%)
  - 结论:caching 无损降本约 40%,永久保留;后续实验以 3/5 为基线
  ```
- ✅ 完成标志:`LOG.md` 有基线记录;能在响应 usage 里指出 `cache_read_input_tokens` 非零。

### Session 4(~2h):手写可靠性四件套
- 工具超时:`execute_tools` 里用 `asyncio.wait_for(fn(), timeout)`,超时把 `"Tool timed out after {N}s"` 作为 tool_result 回传而非抛异常。
- LLM 重试:封装 llm 调用,对 429/5xx/超时做指数退避(`base * 2**attempt + random jitter`),最多 3 次,超限抛出;注意 4xx 客户端错误不重试。
- 费用熔断:loop 里每轮累加费用(单价用 `claude-api` skill 核对),超过 `max_cost_usd` 抛 `CostLimitExceeded`;runner 把它记为 fail(reason: cost)。
- 错误兜底:`run.py` 里 try/except 包住单 case,崩溃记为 error,继续下一个。
- 验证(两条路径都要亲眼看到):写一个 sleep 超长的 mock 工具触发超时;把费用上限临时设 $0.01 触发熔断。
- ✅ 完成标志:超时/熔断均可复现触发且行为符合预期;正常 eval 全量跑完,无一 case 因异常中断整批。

### Session 5(~2h):评估驱动迭代(≥3 轮实验)
- 每轮:改**一个**变量 → 重跑 eval → 记 `LOG.md`。候选变量:
  - system prompt 加"改完必须跑测试确认"的强约束;
  - `edit_file` 的 description 补充"查找串必须唯一匹配"的要求;
  - 工具报错措辞从干巴巴的 error 改成"下一步建议"式(如"匹配到 N 处,请提供更长的上下文片段")。
- 变差的实验也要记,结论写"回滚"。
- ✅ 完成标志:`LOG.md` 累计 ≥3 条"改动 → 通过率变化"记录(基线那条不算),至少 1 条带明确的保留/回滚结论。

### Session 6(~2h):【P1】扩到 10 个 case + runner 体验优化
- 让 Claude Code 再生成 5 个 case,类型上和前 5 个错开(如多文件重构、依赖注入 bug、需要 grep 定位的深层 bug),同样过"红→绿"审核。
- `run.py` 加两个开关:
  - `--case <name>`:单跑一个,调试用;
  - `asyncio.gather` 并发跑多 case(各自独立临时目录,天然无冲突;注意并发度别把 API 限流打爆,3–4 并发起步)。
- ✅ 完成标志:10 case 全量并发跑完出报告,总耗时明显低于串行;`--case` 能单跑指定用例。

## 4. 自测题 + 答案要点

1. **为什么不能靠「感觉 agent 变聪明了」来判断优化是否有效?**
   要点:改动影响非线性,可能这类任务变好那类变差;单次运行有采样随机性,主观印象放大幸存者偏差;只有固定用例集 + 通过率数字才可比较、可复现、可归因;这也是"评估驱动开发"区别于"玄学调 prompt"的分水岭。

2. **一个好的 eval case 需要哪三个组成部分?**
   要点:fixture(独立小仓库,跑时复制到临时目录防污染)、task(任务描述,难度可通过提示粒度调节)、验证方式(客观命令如 pytest 退出码,绝不采信 agent 自述);可补充:好 case 要有区分度,全过/全挂的 case 无信息量。

3. **指数退避(exponential backoff)解决的是什么问题?**
   要点:限流/过载时立即重试会加剧拥堵形成重试风暴;指数级拉长间隔给服务端恢复窗口;加随机 jitter 防止多客户端同步重试;必须有次数上限和最终失败兜底,否则变成慢速死循环;另外要区分可重试错误(429/5xx/超时)与不可重试错误(4xx)。

## 6. 常见坑速查

- **fixture 之间互相污染(agent 改了 A case 的文件影响 B)**。怎么发现:同一 case 第二遍跑结果诡异、fixture 原始目录里出现 agent 的改动。怎么解决:每个 case 复制到 `tempfile.mkdtemp()` 独立目录再跑,跑完即弃。
- **验证方式写得太松(只看 agent 说「我完成了」)**。怎么发现:人工抽查"通过"的 case,发现测试根本没跑或 bug 还在。怎么解决:验证一律用客观命令退出码;审核 fixture 时先确认"修复前必 fail"。
- **eval 跑得慢导致懒得频繁跑**。怎么发现:自己开始"攒几个改动一起跑"——归因已经废了。怎么解决:控制 fixture 规模(文件 ≤6)、并发跑 case、`--case` 单跑调试。
- **caching 永不命中**。怎么发现:`cache_read_input_tokens` 一直是 0。怎么解决:检查前缀里有没有动态内容(时间戳/cwd/随机 id),tools 列表顺序是否稳定,前缀是否达到最小可缓存长度。
- **重试把不该重试的错误也重试了**。怎么发现:400(如 schema 错误)被退避重试 3 次白白等 7 秒。怎么解决:只对 429/5xx/网络超时重试,4xx 类客户端错误直接抛。

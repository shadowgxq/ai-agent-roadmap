# W5｜Workflow 编排(手写,不用框架)

> **一句话目标**:本周结束时,你手写实现了 3 种 workflow 模式(共享 state 的 chaining、routing、evaluator-optimizer)并用自己的 eval 数据说清"什么时候用 workflow、什么时候用 agent";另外 2 种模式(parallelization、orchestrator-workers)能对面试官讲清概念与适用场景。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W4(evals 基线已建立——本周所有"有效性"结论都要靠它出数字)

## 1. 核心要掌握(能口头讲清才算过)

### Workflow vs Agent 的本质区别
- 一句话版本:**workflow 的控制流写死在代码里,agent 的控制流交给模型**。
- 展开:workflow 每一步做什么、下一步去哪是开发者预先决定的,LLM 只在节点内被调用一次干确定的活——可预测、可控、便宜、好调试;agent 是循环里模型自己决定调什么工具、什么时候结束——灵活但不可预测、更贵、更难调试。
- 选型原则:任务路径能预先枚举就用 workflow;路径必须运行时探索(如开放式修 bug)才用 agent。
- 追问点:两者不是二选一——真实系统常是 workflow 骨架里嵌 agent 节点(你 Session 3 就会这么做:E-O 循环里的 optimizer 是整个 agent 一次运行)。

### 共享 state 的 Prompt chaining(本周手写)
- 把大任务拆成固定串行步骤,每步一次 LLM 调用,通过共享的 `WorkflowState`(Pydantic model)传递中间产物(task → plan → code → summary)。
- 价值一:每步 prompt 只做一件事,比一个大 prompt 稳。
- 价值二:步骤间可以插**确定性检查**(gate),如 plan 为空直接失败,不浪费后续调用——这是 workflow 独有的、agent 做不到的确定性控制点。
- 和"多轮对话"的区别:chaining 每步是独立请求,上下文由你从 state 里挑选组装,而不是无脑携带全部历史。

### Routing(本周手写)
- 先用一次便宜调用给输入分类,再路由到不同的专用 prompt/模型配置。
- 核心洞察:**分类是简单任务,必须用最便宜的模型档位**(Haiku 档);路由后简单任务也走小模型——这是成本工程的第一课(W10 深化)。
- 分类输出必须结构化(复用 W2 的结构化输出 + 校验重试);枚举类别里永远留一个 `other` 兜底,否则边界输入会被硬塞进错误分支。

### Evaluator-optimizer(本周手写,重点)
- 循环结构:生成 → 独立评审 → 不通过则带评审意见重来,直到通过或达轮数上限。
- 设计要点一:**evaluator 必须是独立的一次 LLM 调用、prompt 里只给评审职责**——和 coder 共享上下文会"护短"(模型倾向认可自己参与生成的内容),职责隔离才有对抗性。
- 设计要点二:**必须有 max 轮数上限**(如 3 轮)——两个 LLM 可能永远互相不满意,和 W2 结构化输出重试上限是同一个道理;超限要带最后一版产物降级返回,不是直接炸。
- 适用:有清晰评审标准、且迭代确实能改善的任务(代码 review、文案打磨);不适用:评审标准说不清的任务(评分会漂,循环变成噪声)。

### Parallelization(能讲清即可,不写代码)
- 把任务拆成可并行子任务同时跑再聚合,两个变体:**sectioning**(按内容切分,如每个文件并行审查)、**voting**(同一任务跑 N 次投票取多数,用冗余换可靠性)。
- 适用前提:子任务彼此独立、无先后依赖;聚合逻辑要提前定义好。
- 为什么本周不写:工程上就是 `asyncio.gather` 包多次 LLM 调用,W2 的单轮多工具并发已覆盖同源机制,概念清楚即可,时间留给前 3 种。

### Orchestrator-workers(能讲清即可,不写代码)
- 一个协调者 LLM **动态**决定怎么拆任务、派给多个 worker,最后综合结果。
- 和 parallelization 的本质区别:切分方式不是代码写死,而是运行时由模型决定——所以它已经站在 workflow 与 agent 的边界上。
- 为什么本周不写:它的核心机制(派生有独立上下文的子执行者)正是 W6 subagent 要亲手写的,届时自然补齐;现在能讲清"动态拆分 vs 静态拆分"即可。

### 「多 agent 常常是过度设计」的判断框架
- 面试高频题,判断链:①任务路径可否枚举?可以 → workflow;②单 agent + 好工具能否解决?能 → 单 agent;③只有当**上下文隔离**或**真正的并行探索**有实测收益时才上多 agent(W6 会用 token 数字验证)。
- 本周结束你要能带真数字说出:"我实测过,X 类任务 workflow 通过率不低于 agent 且成本只有 1/N。"

## 3. 落地执行(按 session 拆分)

> 硬规则提醒:workflow 循环(chaining 骨架、E-O 循环、router 分发)是核心机制,**必须亲手写**;各步骤的 prompt 文案初稿可以让 AI 起草,但你要自己改到位并用 eval 验证。

### Session 1(~2h):WorkflowState + 共享 state 的 chaining
- 新建 `src/workflows/state.py`:定义 `WorkflowState(BaseModel)`,字段至少含 `task`、`plan`、`code`、`review_feedback: list[str]`、`iteration: int`、`status`。
- 新建 `src/workflows/chaining.py`:
  - 实现 `plan_step(state) → implement_step(state) → summarize_step(state)` 三步串行,每步一次 LLM 调用、只读写 state 的相关字段;
  - 步骤之间加一个确定性 gate(如 plan 为空或超长则 fail-fast,不进下一步);
  - 每步打印:步骤名、输入 token、输出摘要——沿用 W3 的可观测性习惯。
- ✅ 完成标志:一条任务(如"给这个函数写单测")走完 3 步,`state.model_dump_json(indent=2)` 打印出每个字段被正确填充;gate 能用人为置空 plan 触发。

### Session 2(~2.5h):手写 evaluator-optimizer 循环
- 新建 `src/workflows/evaluator_optimizer.py`:
  - `coder(state)`:生成/修改代码;
  - `reviewer(state)`:**独立 LLM 调用**,system prompt 只给评审职责,输出结构化 verdict(Pydantic:`passed: bool`、`issues: list[str]`,复用 W2 结构化输出重试);
  - 主循环:不通过则把 issues 写入 `state.review_feedback`、`iteration += 1` 重跑 coder,**最多 3 轮**,超限带最后一版产物 + 未解决 issues 降级返回。
- 关键实现细节:reviewer 的 messages 里只放"任务要求 + 当前代码",**不放** coder 的生成过程——职责隔离要在代码层面兑现,不是嘴上说说。
- 构造一个 coder 首轮大概率出错的任务(如"实现函数,需处理空输入/负数/超长三种边界",首轮 prompt 不强调边界),亲眼看到完整循环。
- ✅ 完成标志:日志出现"生成 → 评审不通过(附具体 issues)→ 带意见重来 → 通过"的完整轨迹;把 reviewer 标准调苛刻能触发 3 轮上限的降级路径。

### Session 3(~2h):用 eval 数据验证 E-O 的价值
- 把 E-O 包到 agent 外面做成"加强模式":agent 跑完 eval case → reviewer 审查(输入:任务 + diff + 测试输出)→ 不通过则把意见作为新一轮 user message 让 agent 再跑,最多 2 轮。
- 注意这里 optimizer 不是单次 LLM 调用而是整个 agent 一次运行——**模式是通用的,节点粒度可大可小**,这一点本身就是面试谈资。
- 从 W4 eval 集中挑基线容易挂的 case 子集,对比"纯 agent" vs "agent + E-O 收尾"的通过率与成本,记入 `evals/LOG.md`(沿用 W4 格式:日期/改动/通过率前后/结论)。
- ✅ 完成标志:`LOG.md` 新增一条对比记录,有通过率和成本两组数字,结论明确(哪类 case 值得开 E-O、哪类纯属烧钱)。

### Session 4(~2h):手写 router
- 新建 `src/workflows/router.py`:
  - 用最便宜档位模型(用 `claude-api` skill 查 ID)对任务做一次分类,输出结构化结果:枚举 `quick_question` / `bug_fix` / `refactor` / `other` + `confidence`;
  - 按类别路由到不同配置:如 `quick_question` → 小模型 + 精简 system prompt + 少量工具;`bug_fix` → 主力模型 + 全量工具;`other` → 默认全量配置兜底。
- 手造 10 条覆盖各类别和边界的任务描述,跑分类,人工核对准确率;打印每次路由决策与分类调用成本(应是主调用的零头)。
- ✅ 完成标志:10 条人造任务分类准确 ≥9;`other` 兜底分支被至少 1 条边界输入命中;分类单次成本打印可见。

### Session 5(~1.5h):router 接入主链路 + 跑数
- 在 `src/cli.py` 加 `--route` 开关:开启后任务先过 router 再进对应配置的 agent loop。
- 全量跑一遍 eval(开 `--route`),对比通过率(不应下降)与平均成本(简单 case 应下降),记 `LOG.md`。
- 若通过率下降,排查是不是某类 case 被误分类到弱配置——这本身就是"router 的收益边界"的一手数据。
- ✅ 完成标志:`LOG.md` 新增 router 前后对比记录;通过率持平或更好,成本有可见降幅(哪怕只有简单 case 受益也算,如实记录)。

### Session 6(~1.5h):概念补齐 + 口头输出
- 写 `docs/workflow-notes.md`(或 README 一节),三部分:
  1. 5 种模式各一段:定义、典型场景、控制流在谁手里;
  2. parallelization / orchestrator-workers 为什么本周不写代码(前者机制 W2 已覆盖,后者核心留给 W6 subagent);

## 4. 自测题 + 答案要点

1. **五种 workflow 模式各自的典型适用场景?**
   要点:chaining——可预先拆成固定串行步骤(大纲→正文→润色),步骤间可插确定性 gate;routing——输入类型多样、各类型有专属最优配置(客服分流、按难度分级省成本);parallelization——子任务独立可并行(sectioning)或用冗余换可靠性(voting);orchestrator-workers——切分方式无法预知、需运行时决定(跨多文件复杂改动、多源调研);evaluator-optimizer——有清晰评审标准且迭代可改善(代码评审、翻译润色)。

2. **workflow 和 agent 的本质区别用一句话怎么说?**
   要点:控制流在代码里(开发者预先编排,LLM 只填节点)vs 控制流在模型手里(模型运行时决定下一步和终止);推论:workflow 可预测/便宜/好测,agent 灵活/贵/难调;加分项:引用自己的 eval 数字,说明某类任务上 workflow 不输 agent 且成本更低。

3. **evaluator-optimizer 里的 evaluator 为什么要用「独立的一次 LLM 调用、只给评审职责」?**
   要点:与生成者共享上下文会"护短"——模型倾向自我一致而非挑错;独立上下文 + 纯评审角色制造对抗性,类似人类 code review 的角色分离;实现层面 reviewer 的 messages 只含任务与产物、不含生成过程;另需轮数上限防两个模型互相不服的无限循环。

## 6. 常见坑速查

- **把所有东西都做成 agent**。怎么发现:简单任务(如"这个函数干嘛的")也在 agent loop 里烧十几轮。怎么解决:router 把简单类别分流到单次调用/精简配置;用 Session 5 的成本数字提醒自己。
- **evaluator 和 coder 共用同一段上下文 → 评审「护短」**。怎么发现:reviewer 通过率异常高,人工抽查发现明显问题没被指出。怎么解决:reviewer 独立 messages、独立 system prompt,只喂任务与产物;再用苛刻化评审标准测试对抗性是否存在。
- **router 用了贵模型**。怎么发现:打印分类调用成本,发现和主调用一个量级。怎么解决:分类锁死最便宜档位;分类 prompt 精简到几百 token。
- **E-O 无限循环 / 两个模型互相不服**。怎么发现:iteration 一直涨,issues 内容开始重复或互相矛盾。怎么解决:max 3 轮硬上限,超限带最后一版产物 + 未解决 issues 降级返回。
- **state 变成垃圾场**。怎么发现:WorkflowState 字段越加越多,某些步骤读了不该读的字段,prompt 混入无关内容。怎么解决:每步显式声明读/写哪些字段(函数签名或注释),prompt 组装只取所需。

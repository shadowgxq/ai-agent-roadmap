# W6｜SubAgent + Checkpoint(+ P1:最小 Web 查看器)

> **一句话目标**:本周结束时,你的主 agent 能派生独立上下文的 subagent 去探索并只收回总结(有 token 对比数字),任务能 Ctrl+C 后 `--resume` 续跑、失败能 git 回滚,eval 扩到 20 case;时间富余则再加一个流式展示执行过程的最小 Web 查看器。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W5(workflow 三模式已手写;W4 evals 与 LOG.md 习惯已建立;W1 的 SSE 接口将在 P1 复用)

## 1. 核心要掌握(能口头讲清才算过)

### SubAgent 的核心价值 = context 隔离(不是并行)
- 面试第一追问点。子 agent 有**自己独立的 messages** 和受限工具集,去完成子任务(如"探索这个目录并总结"),**只把最终总结返回主 agent**。
- 探索过程中读文件、grep 产生的几万 token 全部留在子上下文里,随子 agent 结束丢弃,不污染主循环。
- 没有隔离会怎样:一次全库探索就能吃掉主 agent 大半 context,后续推理质量随上下文膨胀下降,成本随每轮携带全量历史飙升。
- 并行只是顺带的好处(多个 subagent 可同时跑),隔离才是本质。
- 加分对比:W7 的 auto-compact 是对同一问题的另一种解法(事后压缩),subagent 是事前隔离——两者互补。

### spawn_subagent 的工具化设计
- subagent 对主 agent 而言**就是一个普通工具**:主模型发起 `tool_use: spawn_subagent(task=...)`,你的实现里新建一份 messages、复用同一个 loop 函数跑到结束、把最终文本作为 tool_result 回填。
- 设计决策一:**受限工具集**——探索型 subagent 只给只读工具(read/grep/list),既安全又让它专注。
- 设计决策二:**独立的 system prompt**——明确"你是探索助手,最后必须输出 ≤500 token 总结";输出长度约束是隔离收益的保证,没有它 subagent 会把长篇过程倒回主上下文。
- 设计决策三:**更小的 max_turns**——子任务失控的爆炸半径要小。
- 工具 description 决定主模型会不会用、什么时候用它(W2 的老教训,本周会再踩一次)。

### Checkpoint:messages + state 的序列化
- 每轮循环把 `{run_id, task, messages, turn, 累计费用, workdir}` 完整序列化到磁盘;`--resume` 时反序列化恢复后从断点继续。
- 难点在**保真**:messages 的 content 是结构化 block 列表(text / tool_use / tool_result),不是纯字符串,序列化和恢复要逐字段还原;tool_use 与 tool_result 的配对断裂会直接让下一次请求 400。
- 漏掉任何 state 都会导致 resume 后行为不一致——比如漏了累计费用,熔断阈值形同虚设。
- 追问:checkpoint 除了断点续跑还支撑什么?——任务历史列表、执行过程回放(本周 P1 查看器的数据源)、失败现场分析;它是"把 agent 运行变成可持久化数据"的那一步。

### 失败回滚:用 git 快照,不要自制文件备份
- agent 改坏了要能一键撤销。自制"改前备份文件"方案边界情况极多:新建的文件、删除的文件、目录变更,每种都要单独处理。
- git 快照一条链路全覆盖:任务开始时确保有干净起点 commit(fixture 临时目录没有仓库就 `git init` + 初始 commit),失败时 `git reset --hard <start_sha> && git clean -fd` 回到任务前状态。
- 理解这一点,也就理解了为什么真实 coding agent(Claude Code 等)都深度依赖 git。

### token 对比:让隔离收益变成数字
- "subagent 有用"必须用数字说话:同一个探索型任务,禁用/启用 subagent 各跑一遍,统计**主 agent 每轮请求的 input token 累计值**(usage 逐轮加总)。
- 启用后主上下文只多了一条几百 token 的总结,而不是几万 token 的探索轨迹。
- 注意 trade-off:总 token(主+子)可能不降反升——买到的是主上下文的干净和更长的任务续航,不是省钱。这个 trade-off 要能主动讲出来,数字进 LOG.md,也是简历素材。

### 【P1】最小 Web 查看器的定位
- 骨架早上、皮肤留给 W11–12:复用 W1 的 SSE + 本周的 checkpoint 数据,**一个页面**流式展示 agent 执行过程(文本、工具调用、结果)。
- 明确不做:diff 视图、历史列表、部署。做不完顺延 W7,不挤占 P0。
- 收益:从此 debug agent 不用盯 CLI 日志;W10 开始投递时招牌 demo 已有雏形而不是零。
- 这也是"AI 边界"的示范:SSE 事件 schema 和后端接入自己写(核心机制),页面 UI 放开让 AI 生成。

## 3. 落地执行(按 session 拆分)

> 硬规则提醒:subagent 机制、checkpoint、回滚是核心机制,**必须亲手写**(AI 只答疑);P1 查看器的前端页面 UI 属于"放开让 AI 干"的范围,但 SSE 事件 schema 与后端接入要自己定、自己写。

### Session 1(~2.5h):手写 spawn_subagent
- 前置重构:把 W3 的 loop 函数改造为可参数化(接受 messages / tools / system / max_turns,而非全局单例)——这是本 session 的主要工作量,也是子 agent 复用主循环的前提。
- 新建 `src/tools/subagent.py`,实现 `spawn_subagent(task: str) -> str`:
  - 新建独立 messages(只含 subagent 的 system prompt + task);
  - 只读工具集:`read_file` / `list_dir` / `grep`;
  - `max_turns=15`;复用同一个 loop 函数跑到结束,只返回最终文本。
- subagent 的 system prompt:身份 = 探索助手、只读、最后必须输出 ≤500 token 的结构化总结(发现了什么/关键文件/结论)。
- 注册进工具注册表,description 写清适用场景("需要大量阅读/搜索才能回答的子问题,如摸清一个模块的结构")与不适用场景("单个文件的小改动不要用")。
- ✅ 完成标志:给主 agent 一个"总结 src/ 各模块职责"的任务,日志里看到它调用 `spawn_subagent`、子 agent 独立跑若干轮、主 agent 拿到总结后继续;子 agent 的中间轮次**没有**出现在主 messages 里。

### Session 2(~1.5h):token 对比实验
- 在 loop 里加逐轮 usage 统计:每轮 input/output token 累加,按主/子 agent 分开记。
- 实验任务:"分析整个项目结构并写一份 SUMMARY.md"。跑两遍:
  - A:禁用 spawn_subagent(从注册表摘除);
  - B:启用。
- 对比三个数字:主 agent 累计 input token、总费用(主+子)、任务是否完成。结果记入 `evals/LOG.md`(沿用 W4 格式),如实记录"总 token 可能上升但主上下文显著下降"的 trade-off。
- ✅ 完成标志:LOG.md 有一条带具体数字的对比记录,主 agent 累计 input token 显著下降(量级差可见)。

### Session 3(~2h):手写 checkpoint + --resume
- 新建 `src/agent/checkpoint.py`:
  - 定义 `Checkpoint(BaseModel)`:`run_id`、`task`、`messages`、`turn`、`total_cost`、`workdir`、`start_sha`(Session 4 用);
  - 每轮循环末尾 `model_dump_json` 写入 `.agent-mini/runs/<run_id>/checkpoint.json`;
  - **原子写**:先写临时文件再 `os.rename`,防 Ctrl+C 时写半截。
- `src/cli.py` 加 `--resume <run_id>`:加载 checkpoint、还原 messages 与累计费用、从 `turn` 继续循环。
- 重点自测 content block 保真:恢复后的下一次请求不能因 tool_use/tool_result 配对丢失而报 400。
- 验证:跑一个多轮任务,中途 Ctrl+C,`--resume` 后任务完成;对比日志确认没有重复执行已完成的轮次。
- ✅ 完成标志:Ctrl+C 后 `--resume` 续跑成功;checkpoint.json 里 messages/费用/轮数完整;费用熔断在 resume 后依然基于累计值生效。

### Session 4(~1.5h):git 快照回滚
- 任务启动时(`src/agent/loop.py` 或独立 `snapshot.py`):
  - 工作目录不是 git 仓库 → `git init && git add -A && git commit -m "agent-start"`;
  - 是仓库 → 记录当前 `HEAD` sha;工作区不干净则先自动 commit 一个快照;
  - `start_sha` 存入 checkpoint。
- 提供回滚:`--rollback <run_id>` 从 checkpoint 取 workdir 和 start_sha,执行 `git reset --hard <start_sha> && git clean -fd`;任务失败(MaxTurns/熔断)时打印提示可回滚。
- 验证:把费用上限设极低,让 agent 改了几个文件后熔断,回滚后 `git status` 干净、与开始前 diff 为空。
- ✅ 完成标志:回滚后工作区与任务开始前完全一致(含 agent 新建的文件被 clean 掉)。

### Session 5(~2h):eval 扩到 20 case
- 让 Claude Code 再批量生成约 10 个 fixture(继续走 W4 的"红→绿"审核流程),其中 **2–3 个是需要 subagent 的复杂用例**:多目录、8–15 个文件,任务形如"理解整个项目的数据流后修复 X"——单靠主 agent 硬读会明显吃 context。
- 全量跑 20 case,记录通过率与成本进 LOG.md。
- 检查复杂用例的日志:agent 是否真的用了 spawn_subagent?没用就回去改工具 description 或 system prompt 再跑——这本身又是一条标准的 W4 式实验记录。
- ✅ 完成标志:`run.py` 一键跑完 20 case 出报告;至少 1 个复杂用例的轨迹里出现 spawn_subagent 调用;LOG.md 新增本周通过率记录。

### Session 6(~2.5h):【P1】最小 Web 查看器
- 范围锁死:**单页面、流式展示、不做 diff / 不做历史列表 / 不部署**。
- 后端(自己写):
  - FastAPI 加 `POST /runs`(body: task,启动 agent 任务,返回 run_id)和 `GET /runs/{run_id}/events`(SSE);
  - 自己用 Pydantic 定死事件 schema:`event: text | tool_call | tool_result | done`,data 为 JSON;
  - loop 每步把事件推入 `asyncio.Queue`,SSE 端点消费该队列(复用 W1 的 SSE 写法);连接建立时先从 checkpoint 回放已发生的步骤,再接实时流。
- 前端(放开让 AI 写):一个静态 HTML/JS 页面——EventSource 消费 SSE,按事件类型渲染(文本流式追加、工具调用显示名字+参数摘要+结果摘要),让 Claude Code 生成后你只调交互细节。
- ✅ 完成标志:浏览器打开页面输入任务,能实时看到 agent 每一步(文本/工具调用/结果),任务结束收到 done;中途刷新页面能从 checkpoint 回放已完成的步骤。

## 4. 自测题 + 答案要点

1. **subagent 最主要解决什么问题?(提示:不是「并行」,是「context 隔离」)**
   要点:探索型子任务产生大量中间 token(读文件、grep 结果),放主上下文会稀释注意力、推高每轮成本、加速逼近 context 上限;subagent 用独立 messages 承接这些消耗,只回传压缩总结;并行是附带收益;加分:与 auto-compact 对比——一个事前隔离、一个事后压缩,互补。

2. **为什么「探索型任务」特别适合丢给 subagent?**
   要点:探索的特征是"过程 token 巨大、有价值的结论很小"(读 50 个文件,结论 300 token),压缩比极高,隔离收益最大;且探索只需只读工具,受限工具集安全又聚焦;反例:单文件小修改过程本来就短,派 subagent 反而多付一份 system prompt + 工具定义的固定开销。

3. **checkpoint 除了断点续跑,还能支撑哪些产品功能?**
   要点:任务历史列表(W11–12 Web 版功能)、执行过程回放(本周 P1 查看器刷新后回放)、失败现场分析(带完整 messages 复盘)、费用审计追溯;本质:checkpoint 把 agent 运行变成可持久化数据,一切围绕"运行记录"的产品功能都以它为数据基础。

## 6. 常见坑速查

- **subagent 把完整过程返回主 agent → 失去隔离意义**。怎么发现:主 agent 的 input token 在 subagent 返回后暴涨,和禁用时差不多。怎么解决:subagent 的 system prompt 强制"最后输出 ≤500 token 总结";实现上只回传最终文本,绝不回传子 messages。
- **checkpoint 序列化不全(漏了 state)→ resume 后行为不一致**。怎么发现:resume 后费用从 0 计、或下一次请求因 tool_use/tool_result 配对断裂报 400。怎么解决:Checkpoint 用 Pydantic 全量建模(messages 原样含 content blocks、turn、total_cost、workdir、start_sha),resume 后打印恢复摘要人工核对一次。
- **回滚用文件备份而非 git → 边界情况多**。怎么发现:回滚后残留 agent 新建的文件、或被删文件没恢复。怎么解决:一律 git 快照,`reset --hard` + `clean -fd` 全覆盖;fixture 临时目录先 `git init`。
- **Ctrl+C 恰好打断 checkpoint 写入 → 文件写半截**。怎么发现:resume 时 JSON 解析报错。怎么解决:原子写(临时文件 + rename)。
- **主模型不主动用 spawn_subagent**。怎么发现:复杂用例里主 agent 自己硬读几十个文件。怎么解决:改工具 description(写明适用场景)和 system prompt(建议"全局理解类子任务派给 subagent"),重跑 eval 验证——这就是一条标准的 W4 式实验。
- **(P1)SSE 事件没定 schema 就开写前端 → 渲染混乱**。怎么发现:前端 if/else 越写越乱,某些事件解析不出来。怎么解决:先用 Pydantic 定死事件模型(type + payload),前后端共用这份约定,再让 AI 生成 UI。

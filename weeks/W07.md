# W7｜上下文工程(auto-compact)+ 轻量 RAG 对照实验

> **一句话目标**:agent-mini 在超长任务中能自动压缩历史而不跑偏,并且你能拿出一组自己跑出来的数据回答「RAG vs agentic search 怎么选」。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W3–W6(agent loop、checkpoint、20 case eval 集均已就位)

## 1. 核心要掌握(能口头讲清才算过)

### 上下文工程(context engineering)
- 不是「写好 prompt」,而是把 context window 当成**稀缺资源做预算管理**,目标是「最小充分集」而不是「尽量多」。
- 四类手段:压缩(compact 历史)、检索(RAG / agentic search 按需取)、裁剪(工具输出截断、丢弃无用轮次)、结构化(关键信息固定在 system prompt 或独立消息里)。
- 面试追问点:为什么塞更多上下文往往让模型更差——注意力被稀释、无关信息诱导幻觉、成本随 input token 线性上涨。
- 你在 W3 做的「shell 输出截断」、W6 的 subagent 隔离,本质都是上下文工程,本周把它们串成体系。

### auto-compact(历史自动压缩)【本周 P0 核心,必须亲手写】
- 机制:当累计 token 接近 context window 阈值(如 70%),用一次独立 LLM 调用把旧历史总结成一条 summary 消息替换原文,只保留最近 N 轮——这正是 Claude Code 的 auto-compact。
- 设计点一,触发时机:基于 API 返回的 `usage` 累计或 count_tokens 端点精确计算,不能拍脑袋按消息条数估。
- 设计点二,summary 必须保留:原始任务目标、已完成/未完成事项、关键文件路径与修改记录、当前卡在哪。丢了任务目标,agent compact 后必跑偏。
- 设计点三,消息序列合法性:compact 后的 messages 不能出现悬空 tool_use(有 tool_use 无对应 tool_result,下一轮请求直接 400),切割点必须落在完整轮次边界。
- 追问到实现也要能答:compact 用小模型即可(总结不需要主力模型);compact 失败要有降级路径,不能把主任务拖挂。

### compact 与 prompt caching 的交互
- prompt caching 靠「前缀逐字节一致」命中;compact 重写历史等于换前缀,会话级缓存必然失效一次(system prompt + 工具定义在最前,不受影响)。
- 因此阈值不能设太低:频繁 compact = 频繁 cache miss + 频繁多付一次总结调用,这是典型成本 trade-off,要用数据选阈值。

### RAG(检索增强生成)
- pipeline:文档切块(chunking)→ embedding 向量化 → 存入向量库 → 查询时算相似度取 top-k 塞进 prompt。
- chunking 粒度 trade-off:太大→检索粒度粗、单块噪音多;太小→语义被切碎、半个函数没法用。代码场景按函数/类等语法边界切,不按固定字符数硬切。
- 余弦相似度是默认度量(只关心方向不关心模长);top-k 与相似度阈值都要实验调,不是常数。
- 本周只用 `sqlite-vec` 或内存 numpy;pgvector 等生产级向量库到「面试能一句话讲清定位差异」即可(主文档第 10 节口径)。

### RAG vs agentic search【高频面试题】
- RAG = 预建索引 + 相似度检索:一次调用拿候选,便宜、快,但受 chunking 质量限制,且「语义相似 ≠ 逻辑相关」。
- agentic search = agent 用 grep/read 自己找:能顺着线索多跳(先 grep 调用点、再 read 定义),对代码任务往往**更准**;代价是多轮工具调用,更贵更慢。
- 结论不能背,要用 Session 5–6 自己的数据说话:什么任务 RAG 够了、什么任务必须 agentic search。

### Agent memory(跨会话记忆,只讲概念,不实现)
- 三个易混概念:checkpoint = **同一任务内**断点续跑(序列化 messages+state,W6 已做);RAG = **外部知识**检索;memory = **跨会话**经验沉淀。
- memory 的典型实现:会话结束时用 LLM 提炼「用户偏好、踩过的坑、项目约定」写入持久存储(常见就是 markdown 文件),下次会话启动时注入 context。
- 联系:memory 的写入端复用「LLM 总结」能力(同 compact);读取端本质是检索——小规模直接全量注入,大规模退化为 RAG。
- 近年面试出现频率在涨,要求:能对着面试官讲清这张区别联系表,3 分钟不卡壳。

## 3. 落地执行(按 session 拆分)

> 硬规则提醒:auto-compact 是主文档钦定「必须亲手写」的四大核心机制之一——Session 2/3 的代码 AI 只答疑、不代写;Session 5–6 的实验语料和脚手架可放开让 AI 干。

### Session 1(~2h):精读主教材 + 摸清自己 agent 的 token 现状
- 精读 Effective context engineering 全文,把「compaction 要保留的信息清单」抄进笔记。
- 给 `src/agent/context.py` 加 `token_usage()` 方法:基于每次 API 响应的 `usage.input_tokens` 累计,或调 count_tokens 端点精确计算当前 messages 总量(含 system prompt 与工具定义)。
- 挑一个 W6 eval 里最长的 case 跑一遍,CLI 每轮打印 `turn=N tokens=X (Y% of window)`。
- 观察 token 增长曲线,确认哪类工具输出是大头(通常是 `run_shell` 和 `read_file`)。
- 在笔记里写下你的 compact 设计方案:阈值、summary 保留清单、切割点规则。
- ✅ 完成标志:CLI 每轮输出 context 占用百分比;compact 设计方案笔记完成。

### Session 2(~2.5h):手写 auto-compact v1
- 新建 `src/agent/compact.py`,实现 `async def compact(messages, keep_recent: int = 4) -> list`:
  - 按轮次边界切出「待压缩段」,确保不拆开 tool_use/tool_result 配对;
  - 用一次独立 LLM 调用(小模型档)生成结构化 summary,prompt 明确要求输出:原始任务、已完成事项、修改过的文件及要点、未完成事项、当前状态;
  - 返回 `[summary 消息] + 最近 keep_recent 轮原文`。
- 在 `loop.py` 每轮开始处检查 `token_usage() > threshold * CONTEXT_WINDOW`(默认 0.7)则触发,打印 `[compact] before=X tokens, after=Y tokens`。
- 阈值、keep_recent 进 `config.py` 可调。
- ✅ 完成标志:构造人为长任务(如让 agent 逐个 read 一批大文件),日志可见 compact 触发且 token 明显下降,后续请求无 tool_use/tool_result 配对错误。

### Session 3(~2h):关键信息保护 + 长任务实测
- 强化「不能丢」:两种做法都试——①第一条 user 消息永不进压缩段;②summary prompt 强制原文复述任务目标。选实测表现好的。
- 实测 2 个真实长任务(如「重构 X 模块并保证测试通过」),人工检查 compact 后 agent 是否仍朝原目标推进、是否重复读已读过的文件。
- 边界处理:compact 的 LLM 调用失败时降级——重试一次,再失败则硬截断最旧轮次保命,不让 compact 把主任务搞挂。
- 压力测试:故意把阈值调到 30% 制造一个任务内多次 compact,验证不跑偏。
- ✅ 完成标志:2 个长任务在触发 compact 后正确完成;30% 阈值多次 compact 仍不跑偏。

### Session 4(~1.5h):eval 回归 + LOG 记录 + memory 概念梳理
- 全量跑 `python evals/run.py`(20 case),确认 compact 上线没有拉低通过率。
- 对比开/关 compact 的平均单 case 费用:compact 多一次总结调用、但省长尾 input token,看数据下结论。
- `evals/LOG.md` 记一条:「加入 auto-compact:通过率 X→Y,长 case 费用变化,阈值=70%,keep_recent=4」。
- 花 30min 写 `notes/memory-vs-checkpoint-vs-rag.md`:三者区别联系表 + 各一个使用场景,按第 1 节口径整理。
- ✅ 完成标志:LOG.md 新增 compact 实验记录;笔记完成且能脱稿口述 3 分钟。

### Session 5(~2h):【P1】最小 RAG——chunking + embedding + 余弦检索
- 新建 `src/rag/`:`indexer.py` + `retriever.py`。
- 索引器:遍历目标代码库,按「函数/类为单位、上限 N 行」切块(简单做法:按 `def`/`class` 行分段,别在函数中间硬切),每块记录文件路径 + 行号;embedding 用选定方案。
- 存储二选一:`sqlite-vec`(照 README 的 Python 示例建虚拟表 + KNN 查询)或最简内存版(numpy 矩阵,余弦相似度 `(A @ q) / (norm(A, axis=1) * norm(q))`)。
- 检索器:`retrieve(query, top_k=5)` 返回块内容 + 来源位置;写 10 行 CLI 冒烟脚本,对某开源中型仓库索引后问「X 功能在哪实现」,肉眼确认 top-5 命中。
- 语料与脚手架可让 AI 代写,但每一步在干什么你必须能解释。
- ✅ 完成标志:对 ≥100 文件的仓库建索引 <1min、单次检索 <1s,top-5 能命中正确文件。

### Session 6(~2h):【P1】RAG vs agentic search 对照实验
- 设计 8–10 个「在大代码库里回答问题」的 QA case(如「哪个函数处理了 Y?给出文件和函数名」);case 生成可 AI 辅助,标准答案必须亲自核对。
- 两组对照:A 组 agent 只挂 `rag_search` 工具;B 组只有 grep/read(现有 agentic search)。
- 每组记录三个指标:正确率、平均 token 成本、平均耗时。
- 结果与结论写进 `evals/LOG.md`:哪类问题 RAG 赢(模糊语义式提问)、哪类 agentic search 赢(需要多跳追踪的),给出你的选型判断,并对照主教材的论述看是否一致。
- ✅ 完成标志:LOG.md 有两组对照数据表 + 3 句以内结论。

## 4. 自测题 + 答案要点

1. **auto-compact 的触发时机和实现方式?压缩时要保留哪些信息不能丢?**
   要点:累计 usage 或 count_tokens 达阈值(如 70%)触发;独立 LLM 调用把旧历史总结为 summary 替换原文,保留最近 N 轮;不能丢——原始任务目标、已改文件清单、未完成事项、当前进展;切割必须保持 tool_use/tool_result 配对完整。

2. **什么时候用 RAG,什么时候 agentic search 就够了?**
   要点:代码任务且 agent 有文件工具时 agentic search 通常更准(可多跳、拿到完整源码上下文);RAG 适合超大语料一次性语义定位、无工具环境、延迟/成本敏感的单跳问答;引用自己 Session 6 的数据佐证——这是「有数据的回答」与背书答案的区别。

3. **chunking 的粒度太大 / 太小分别有什么问题?**
   要点:太大→单块噪音多、top-k 总量爆、相似度被无关内容稀释;太小→语义完整性破坏(半个函数)、命中了也不可用、块数暴涨拖慢索引;代码场景按函数/类语法边界切。

4. **(补充)compact 和 prompt caching 是什么关系?**
   要点:caching 要求前缀逐字节一致,compact 改写历史必使会话级缓存失效一次;system prompt + 工具定义前缀不受影响;阈值过低导致「频繁失效 + 频繁总结调用」双重浪费,用成本数据选阈值。

5. **(补充)Agent memory 和 checkpoint、RAG 的区别与联系?**
   要点:checkpoint=同一任务内状态快照(续跑/回滚);RAG=外部知识按需检索;memory=跨会话经验沉淀(结束时 LLM 提炼写入、启动时注入);联系——写入端复用总结能力(同 compact),读取端大规模时退化为 RAG。

## 6. 常见坑速查

- **压缩时把「当前任务目标」也压没了 → agent compact 后跑偏**(主文档坑)。
  - 怎么发现:compact 后 agent 答非所问、重新探索已完成的部分。
  - 怎么解决:第一条 user 消息永不进压缩段,或 summary prompt 强制复述原始任务。
- **切割点拆开 tool_use/tool_result 配对 → 下一轮 API 直接 400**。
  - 怎么发现:compact 后第一次请求报 `tool_use ids were found without tool_result blocks` 类错误。
  - 怎么解决:只在「assistant 纯文本回复结束」的完整轮次边界切割。
- **RAG chunk 边界切在函数中间 → 检索到半个函数没用**(主文档坑)。
  - 怎么发现:抽查 top-5 结果,内容以残缺代码开头/结尾。
  - 怎么解决:按函数/类语法单位切块。
- **盲目上 RAG → 很多场景 agentic search 更简单更准**(主文档坑)。
  - 怎么发现:对照实验里 RAG 组正确率不升反降,还多了索引维护成本。
  - 怎么解决:默认 agentic search,RAG 只在 Session 6 数据证明它赢的任务类型上启用。
- **compact 阈值设太低 → 频繁总结调用 + cache 反复失效,成本反升**。
  - 怎么发现:Session 4 费用对比里开 compact 反而更贵。
  - 怎么解决:阈值回调到 70% 左右,对比不同阈值的费用数据再定。

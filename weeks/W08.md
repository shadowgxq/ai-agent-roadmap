# W8｜MCP:client 架构重构 + 开源自己的 MCP server

> **一句话目标**:agent-mini 能同时使用本地工具和任意外部 MCP server 的工具,并且你在 GitHub 上有一个能被 Claude Code/Desktop 直接配置使用的、带前端领域差异化的开源 MCP server(第二交付物)。
> **时间预算**:10–15h(约 5–6 个 session,每个 1.5–2.5h)｜**前置**:W2(工具注册表)、W3–W7(agent-mini 主线)

## 1. 核心要掌握(能口头讲清才算过)

### MCP 是什么、解决什么重复劳动
- Model Context Protocol:标准化「模型/agent 如何连接外部工具和数据源」的开放协议。
- 核心论证是 M×N 问题:M 个 AI 应用 × N 个数据源,无协议时每对组合都要写一次集成;有 MCP 后工具方写一次 server、应用方写一次 client,M×N 变 M+N。
- 「AI 界的 USB-C」的准确含义:不是比喻它快,是比喻**接口标准化消灭适配层**。
- 面试追问点:MCP 和 tool use 什么关系——tool use 是模型层的调用机制,MCP 是应用层的分发/接入协议;MCP tool 最终仍以 tool use 的形式呈现给模型。

### 三大原语:tools / resources / prompts
- `tools`:模型可主动调用的函数——**model-controlled**,模型决定何时调(如「跑一次性能审计」)。
- `resources`:可读取的数据/上下文,如文件、日志、schema——**application-controlled**,宿主应用决定注入什么(如「上次审计报告」)。
- `prompts`:预置提示模板——**user-controlled**,用户显式触发(如 `/audit 这个页面`)。
- 区分的本质是**控制权在谁手里**——最容易被追问、也最容易答混的一题。
- 多数第三方 server 只实现 tools;你的 server 合理用上 resources/prompts 是加分项。

### client / server 架构与 transport
- server 暴露能力;client(Claude Desktop、Claude Code、你的 agent-mini)连接并使用。
- transport 两条主路:`stdio`(client 把 server 当子进程拉起,本地默认,配置就是一条启动命令)和 `streamable HTTP`(远程部署)。
- 生命周期:client 启动 → `initialize` 握手 → `tools/list` 发现能力 → 调用时发 `tools/call`。
- 结论:外部工具对你的 agent loop 只是「schema 来源和执行方式不同的工具」,循环本身不用改——这就是好协议的价值。

### FastMCP 编程模型
- 官方 Python SDK 的高层 API:`mcp = FastMCP("name")`,函数加 `@mcp.tool()`,类型注解 + docstring 自动生成 JSON Schema。
- 与你 W2 手写的工具注册表**同构**——「我手写过同款机制,所以知道装饰器背后发生了什么」是现成的面试素材。
- `@mcp.resource("scheme://path")`、`@mcp.prompt()` 对应另外两个原语;SDK 自带 MCP Inspector 做本地调试。

### MCP client 集成到自有 agent
- 命名冲突:不同 server 可能都叫 `search`,并入注册表时加前缀(如 `lighthouse__audit`)。
- schema 归一:MCP 的 `inputSchema` 就是 JSON Schema,与 Claude API 的 tools 参数几乎直通,但要验证可选字段等边角。
- 生命周期:stdio server 是子进程,用 `AsyncExitStack` 管理多 server session,agent 退出(含异常路径)必须干净关闭。

### 开源工程素养
- 一个「能被用起来」的 MCP server = 差异化选题 + 精准 tool description(W2 教训:描述质量直接决定模型调用准确率)+ README + 可安装性(`uvx` 一条命令能跑)。
- 工具输出也要做上下文工程:把底层工具的几万字符原始 JSON 加工成模型友好的结构化摘要,别把 W3 的「输出撑爆 context」问题在协议边界重演。
- 开源不等于 push 代码,等于**别人 5 分钟内能跑起来**。

## 3. 落地执行(按 session 拆分)

> 硬规则提醒:MCP client 接入属于工具层核心架构,「亲手写」范畴(它是 agent 核心回路的一部分);开源 server 的样板(README 骨架、打包配置、CI、测试脚手架)可放开让 AI 干,但 tool description 必须自己反复打磨。

### Session 1(~2h):MCP 概念 + 跑通官方示例
- 精读官方文档 Concepts 各节。
- 画一张「host / client / server + 三大原语」架构图(手画拍照或 mermaid,存 `notes/`)。
- 用 `claude mcp add` 给本机 Claude Code 挂一个官方 server(如 filesystem),真实对话中观察:工具何时被调用、在 `/mcp` 面板里长什么样。
- ✅ 完成标志:架构图完成;Claude Code 里成功调用了一次外部 MCP 工具;能不看文档口述三大原语的控制权区别。

### Session 2(~2h):FastMCP 写最小 server + 吃自己的狗粮
- 新建独立仓库(名字按 Session 4 选题定,先叫 `mcp-server-playground` 也行),`uv init` + 安装 `mcp` SDK。
- 写最小 server:1 个 `@mcp.tool()`(如 `echo`/`add`)+ 1 个 `@mcp.resource()` + 1 个 `@mcp.prompt()`,三大原语各摸一遍。
- 用 SDK 自带的 MCP Inspector 本地调试,确认 schema 生成符合预期。
- `claude mcp add` 配进 Claude Code,真实对话触发调用成功。
- ✅ 完成标志:自己的 server 出现在 Claude Code 的 MCP 列表且工具调用返回正确;三个原语各验证一次。

### Session 3(~2.5h):agent-mini 重构为 MCP client
- 新建 `src/tools/mcp_client.py`:用 SDK 的 client API(stdio transport)连接配置里声明的 server 列表 → `tools/list` 拉取工具清单 → 转换为注册表统一格式并入(名字加 `serverName__` 前缀)→ 执行时路由到对应 session 的 `tools/call`。
- 新增 `mcp_servers.json` 配置(server 名 → 启动命令/args/env)。
- 用 `AsyncExitStack` 管理多 server 生命周期,agent 退出时(含异常)干净关闭子进程。
- 验证:一条任务同时用到本地工具(`read_file`)和外部 MCP 工具(Session 2 的 server),CLI 日志能区分两类工具来源。
- ✅ 完成标志:同一次 agent run 中本地与 MCP 工具混合调用成功;`python evals/run.py` 回归无下降(重构没伤到主线)。

### Session 4(~2h):开源 server 选题定案 + 核心功能 v1
- 四个候选的差异化分析(30min 内做决定,别陷入选择消耗):
  - **`mcp-server-lighthouse`**(网页性能审计):数据最有冲击力(分数、指标一目了然);实现是对 lighthouse CLI 的子进程封装,风险最低;但方向相对热门,差异化中等。适合想稳妥出成果的。
  - **`mcp-server-a11y`**(可访问性检查):基于 axe-core,竞品明显更少;a11y 是有真实痛点且 AI 工具覆盖差的领域,差异化高;需要 headless 浏览器编排,难度中等。适合想要独特性的。
  - **`mcp-server-bundle-analyzer`**(前端产物体积分析):吃透 stats.json / 构建产物是纯前端内功,最能体现「懂前端的人才写得出」;但输入依赖用户的构建产物,demo 门槛略高。适合面试想深聊工程化的。
  - **视觉回归(figma-visual-qa 思路产品化)**:你已有 skill 积累可直接迁移,叙事最完整(「把自己的工作流协议化成了开源工具」);但截图对比的环境依赖(Playwright)最重,工作量最大。适合时间充裕、想要最强故事的。
- 决策建议:时间紧选 lighthouse,求差异化选 a11y;定了就把选题理由写进 README 第一段,不再回头。
- 实现核心 tool v1:输入 URL/路径 → 子进程跑底层工具 → 把原始输出**加工成模型友好的结构化摘要**(关键分数 + top 问题列表 + 修复建议字段),而不是原样返回几万字符 JSON。
- ✅ 完成标志:核心 tool 在 MCP Inspector 里对真实输入返回结构化结果,单次输出 <2k tokens。

### Session 5(~2h):server 完善 + README + 开源发布
- 补齐第二个能力:再加一个 tool(如「对比两次审计结果」)或一个 resource(如「最近一次审计报告」)。

## 4. 自测题 + 答案要点

1. **MCP 的 tools / resources / prompts 分别是什么,区别在哪?**
   要点:tools=模型可调用的函数(model-controlled);resources=可读数据/上下文(application-controlled);prompts=预置模板(user-controlled);核心区分维度是「谁决定何时使用」;举例——审计动作是 tool、上次报告是 resource、`/audit` 是 prompt。

2. **为什么说 MCP 是「AI 界的 USB-C」?它解决了什么重复劳动?**
   要点:M 个应用 × N 个工具源的两两定制集成 → 协议化后各写一次,M×N 变 M+N;工具方一次开发即可被所有 MCP client 复用;与 tool use 的关系:应用层分发协议 vs 模型层调用机制。

3. **你的 agent 作为 MCP client,怎么把外部 server 的工具并入自己的注册表?**
   要点:按配置拉起 server(stdio 子进程)→ `initialize` 握手 → `tools/list` 拿工具及 inputSchema → 加 server 前缀防命名冲突、转成注册表统一条目 → 模型请求时路由到该 session 的 `tools/call`,结果照常作为 tool_result 回填;agent loop 本体零改动。

4. **(补充)你的 server 为什么把底层工具输出加工成摘要,而不是原样返回?**
   要点:原始 JSON 动辄数万字符,直接回填会撑爆调用方 context(W3 输出截断问题在协议边界重演);工具输出是 context 的一部分,server 作者有责任返回「最小充分信息」——结构化分数 + top 问题 + 建议,细节按需再查。

## 6. 常见坑速查

- **tool description 写得含糊 → 模型不会正确调用**(主文档坑,与 W2 教训一致)。
  - 怎么发现:狗粮测试里模型该调不调、或传错参数。
  - 怎么解决:description 写清「何时用我 + 参数语义 + 返回什么」,用真实对话反复迭代。
- **选题太通用(又一个 weather server)→ 没有差异化**(主文档坑)。
  - 怎么发现:GitHub 一搜同名 server 一堆;README 写不出「为什么是我来做」。
  - 怎么解决:锁死在 Session 4 四个前端交叉候选内,选题理由写进 README。
- **忘了写 README / 配置示例 → 开源等于没发**(主文档坑)。
  - 怎么发现:让没参与的人从零装,10 分钟跑不起来。
  - 怎么解决:README 必含 Claude Code + Claude Desktop 两份可复制配置片段,发布前做干净环境验证。
- **stdio server 里往 stdout 打日志 → 破坏 JSON-RPC 消息流**。
  - 怎么发现:server 单独跑正常,被 client 拉起就断连/报协议解析错误。
  - 怎么解决:日志一律走 stderr 或文件,stdout 只留给协议。
- **子进程生命周期没管好 → agent 退出后残留僵尸 server 进程**。
  - 怎么发现:跑几次 eval 后 `ps aux | grep mcp` 里一排残留进程。
  - 怎么解决:`AsyncExitStack` 统一管理,所有退出路径(含异常)确保 close。

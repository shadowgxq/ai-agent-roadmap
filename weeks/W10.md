## 1. 核心要掌握(能口头讲清才算过)

### 权限系统的三层防线
生产级 coding agent 的安全不是单点开关,是纵深防御,三层各拦一类风险:
- **第一层:shell 命令白/黑名单**(能不能执行)——拦「模型调了不该调的命令」。
- **第二层:文件操作范围限制**(能碰哪些路径)——拦「工具参数越界写到工作区外」。
- **第三层:危险操作人在回路确认**(执行前要不要人批准)——拦「命令合法但后果不可逆」(`rm -rf`、`git push --force`)。

面试追问「为什么不只做黑名单」时要能答:黑名单永远列不全(`rm` 拦了还有 `find -delete`、`python -c "os.remove(...)"`),白名单是默认拒绝、按需放行,这正是 Claude Code 权限系统的设计动机。还要能补一句:这三层都是「策略层」,真正的执行隔离(机制层)是 W11–12 的 Docker 容器——策略防误用,机制防恶意。

### 命令解析与绕过风险
白/黑名单不能用「字符串包含」判断——`echo rm` 会被误杀,`ls; rm -rf /` 会被漏放。正确做法:用 `shlex.split` 解析出真正的命令词,再处理复合结构(`;`、`&&`、`|`、`$()` 子命令)——最稳妥的策略是**含复合结构的命令一律降级为「需确认」**,而不是试图完美解析。还要意识到:名单只是护栏不是沙箱,真正的执行隔离要到 W11–12 用 Docker 解决,这周先把「策略层」做对。

### 文件操作范围限制(path sandbox)
所有 fs 工具(`read_file`/`write_file`/`edit_file`)在执行前把路径 `Path.resolve()` 成绝对路径,校验它是否在允许的 workspace root 之下。要点:必须 resolve 之后再比较(否则 `../../etc/passwd` 这类相对路径穿越就漏了);符号链接也要 resolve;拒绝时把清晰的错误信息作为 `tool_result` 回给模型让它自我纠正,而不是抛 500(W2 的老原则)。

### 危险操作「人在回路」确认
哪些算危险,判定标准是「后果可逆性 + 影响范围」:
- 删除类:`rm`、`git clean`、`git reset --hard`。
- 改远端类:`git push`(尤其 `--force`)。
- 装包类:`pip install` / `npm install`(安装脚本可执行任意代码)。
- 网络外发类:`curl` / `wget` 的 POST(数据泄露面)。

实现上是一个 `confirm` 回调:CLI 场景下阻塞等待 y/n 输入;为 W11–12 的 Web 版预留接口(确认请求也是一种事件)。要能讲清:确认的本质是**把不可逆操作的决策权还给人**,agent 的判断可以错,但错误的代价必须可控;同时确认不能滥用——常规操作全弹确认会造成「确认疲劳」,人会无脑点 yes,反而更不安全。

### prompt caching 的命中机制与收益核算
W4 起你已经开着 caching 跑了 6 周,本周要做的是**拿出数据**。必须讲清的机制点:
- 缓存按**前缀完全一致**命中:请求体按 tools 定义 → system prompt → messages 的顺序从头逐段匹配,前缀里任何一个字节变了(比如 system prompt 里嵌了当前时间戳)缓存就全废。
- `cache_control` 标记的是「缓存到此为止的前缀」,不是「缓存这一块」——这是最常见的理解偏差。
- 成本模型:缓存写入比普通 input 贵(约 1.25 倍),读取只要约 0.1 倍,TTL 默认 5 分钟;各档模型有最小可缓存 token 数门槛(具体数值以官方文档为准)。
- 收益公式:**同一前缀被复用的次数越多、前缀越长,收益越大**——agent loop 每轮都带全量历史重发,上一轮的完整请求就是这一轮的缓存前缀,是 caching 的最佳场景;一次性请求反而略亏(白付写入溢价)。

### router 分流的成本收益量化
W5 手写的 router 本周正式接入成本工程:用最便宜档位(Haiku 档)先给任务分类,简单任务(改注释、重命名、单文件小修)走小模型,复杂任务走主力模型。量化方法:eval 集全量跑两遍(分流关/开),对比「总通过率」和「单任务平均成本」两个数字——**通过率不能明显掉,成本要明显降**,只报成本降了多少而不报通过率变化是造数据。

### 技术写作:用数据讲「问题 → 方法 → 结果」
博客不是学习笔记,是工程复盘。三个要点:
- 每一轮优化都要有「改了什么(单变量)→ 数字变化 → 为什么」,数字全部出自 `evals/LOG.md`,可复查。
- 踩坑段落(死循环、context 爆、改完不验证)是全文最有价值的部分——读者和面试官都清楚:没踩过坑的「成功经验」不可信。
- 图表比表格有说服力:一张「实验编号 × 通过率」的折线图,胜过三段文字。

## 4. 自测题 + 答案要点

**1. 为什么危险操作要「人在回路」确认?哪些操作算危险?**
- 模型判断会错,而 `rm`/`git push --force` 这类操作**不可逆**,错误代价不可控,必须把最终决策权还给人。
- 危险的判定标准是「后果可逆性 + 影响范围」:删除类、改远端类、装包类(可执行任意代码)、网络外发类(数据泄露面)。
- 补充:确认不是万能的——确认疲劳会让人无脑点 yes,所以常规安全操作要自动放行,确认只留给真正危险的少数。

**2. prompt caching 的缓存前缀命中条件是什么?放什么内容进去收益最大?**
- 按**前缀完全一致**命中:tools → system → messages 从头逐段匹配,`cache_control` 标记前缀终点;前缀中任何变动(哪怕一个时间戳)导致整段不命中。
- 放「大而不变、每次请求都带」的内容:工具定义 + system prompt;agent loop 里逐轮增长的历史也天然受益(上一轮的前缀就是这一轮的缓存)。
- 成本模型:写入约 1.25 倍、读取约 0.1 倍普通 input 价,TTL 5 分钟——高频复用长前缀才划算,一次性请求反而略亏。

**3. router 分流的成本收益怎么用 eval 数据量化?**
- 固定 eval 集全量跑两遍(分流关/开),对比两个维度:总通过率 + 单任务平均成本。
- 结论必须同时报两个数:成本降幅只有在「通过率不明显下降」前提下才有意义,否则是拿质量换钱。
- 还要把分类器自身的调用成本算进总账;分类错误(复杂任务被派给小模型)会体现在通过率上,这正是要跑全量 eval 而不是抽查的原因。

**4.(补充)白名单/黑名单/需确认三档,一条命令同时命中多档时优先级怎么定?**
- 拒绝优先:命中黑名单直接拒,不给确认机会(黑名单里都是明确不该让 agent 碰的)。
- 其次确认:含复合结构(`;`、`|`、`$()`)或命中确认名单的,统一走人在回路。
- 最后放行:纯命令词在白名单内才自动执行;三档都没命中的未知命令,默认落到「需确认」而不是放行——默认拒绝原则。

**5.(补充)加了权限护栏之后,怎么证明 agent 的能力没有被削弱?**
- 唯一客观方法:eval 集全量回归,对比加护栏前后的通过率(这就是 Session 2 的回归验证步骤)。
- 若通过率下降,看失败 case 的 trace:是白名单拦了正常命令(调名单),还是确认打断了自动化流程(eval 模式下危险操作应直接拒绝并让模型换路径,而不是等人)。
- 这个「护栏 vs 能力」的平衡数据,本身就是博客和面试的好素材。

## 6. 常见坑速查

- **白名单太严 → agent 干不了活;太松 → 有安全风险**。怎么发现:加完护栏 eval 通过率骤降(太严)或危险命令测试能穿透(太松)。怎么解决:基于真实 eval 跑出来的命令使用频率调名单,拿数据找平衡,不拍脑袋。
- **prompt caching 前缀里放了会变的内容 → 缓存永远不命中**。怎么发现:Langfuse/API 返回里 cache_read_input_tokens 恒为 0。怎么解决:检查 system prompt/tools 里的时间戳、随机 id、动态路径,把可变内容移到前缀之后。
- **博客只讲成功不讲踩坑 → 不可信也不吸引人**。怎么发现:通读初稿,若「坑」段落少于全文 1/4 就是报喜稿。怎么解决:回翻 W3–W9 的 LOG.md 和 git 历史,把死循环、context 爆、judge 不稳这些真实翻车现场原样写进去。
- **黑名单用字符串包含匹配 → 误杀 `echo rm` 或漏放 `ls; rm -rf /`**。怎么发现:policy 单测里专门放这两类用例。怎么解决:`shlex` 解析命令词 + 复合结构一律降级为需确认。
- **成本对比实验一次改多个变量 → 无法归因**(W9 老坑重现)。怎么发现:LOG.md 里一条记录同时写了「开了缓存又换了模型」。怎么解决:router 和 caching 分两个独立实验跑,各自留对照组。

# W11-12｜产品化冲刺:Web 版上线

> **一句话目标**:两周结束时,`agent-mini` 有一个**公开 URL**,陌生人打开首页点一个预置 demo,就能实时看到 agent 执行全过程(流式文本、工具调用卡片、文件 diff、任务历史),且 agent 在 Docker 容器内隔离执行;顺带能在面试里讲清 LangGraph 和 Claude Agent SDK 与你手写实现的异同。
> **时间预算**:两周 20–30h(约 10–12 个 session,每个 1.5–2.5h)｜**前置**:W6(checkpoint + P1 最小 Web 查看器雏形)、W10(权限护栏、confirm 回调接口)

> **本周定位提醒**:你是前端专家——Next.js/组件/样式这部分**放开让 AI 辅助写**,那是发挥优势不是学习重点;这两周真正的学习重点是**多类型流式事件协议的设计**(后端怎么把 agent 的执行过程结构化成事件流、前端怎么按类型消费)。W6 的 P1 最小 Web 查看器已经打好骨架,这两周是给它**加皮肤和功能**,不是从零开始。

## 1. 核心要掌握(能口头讲清才算过)

### 结构化 SSE 事件协议设计(本两周的灵魂)
主文档的红字:「SSE 多事件类型没设计好协议 → 前端渲染混乱,先定死事件 schema」。事件类型定死五种:`text` / `tool_call` / `tool_result` / `diff` / `done`。设计要点:
- 每个事件带 `task_id` + 自增 `seq`(前端据此保序、去重、断线续传)。
- `tool_call` 和 `tool_result` 用同一个 `tool_use_id` 关联(前端才能把结果渲染回对应卡片)。
- schema 用 Pydantic 定义并**单独成文件**,它是前后端契约,先冻结再开工;之后的变更必须走版本号。
- `done` 事件携带终态(success/failed/max_turns)和总成本,是前端停止 loading 的唯一依据——不能靠「流断了」推断结束,因为断流也可能是网络故障。

面试追问「为什么不直接把 messages 原样推给前端」要能答:messages 是模型协议不是 UI 协议,含大量前端不需要的细节且结构随 SDK 变化,中间的事件层就是防腐层——这和前端里「API response 不直接进组件、先过一层 view model」是同一个思想。

### Vercel AI SDK 的 useChat / data stream
`useChat` 解决的是「流式对话的前端状态管理」:自动维护 messages、loading、错误重试。关键是它的 **data stream protocol**:除了文本增量,还支持自定义 data parts——你的 `tool_call`/`diff` 事件正是通过 data parts 通道下发,前端按 `type` 分发到不同组件渲染。要能讲清两种接法的取舍:
- 接法①:后端直接实现 AI SDK 的 data stream protocol——前端零胶水,`useChat` 开箱即用,但后端被协议格式绑定。
- 接法②:后端保持自己的 SSE schema,前端用原生 `EventSource`/`fetch` + ReadableStream 自己解析——协议自主,胶水自己写。

本项目推荐②或混合——因为你的事件类型(diff、confirm)超出了标准 chat 场景,协议自主更重要;这个取舍本身就是面试谈资(「我评估过 AI SDK,最终为什么自研消费层」比「我用了 AI SDK」更能体现判断力)。

### Docker 容器隔离(安全红线,不可妥协)
Web 版意味着**陌生人能给你的 agent 下任务**,agent 会执行任意代码——不隔离,等于把服务器 shell 公开给全网。W10 的白名单是「策略层」,容器是「机制层」,策略可被绕过、机制不能省。实现要点:
- 每个任务一个一次性容器(`docker run --rm`),fixture/工作区以 volume 挂进去,任务结束即销毁。
- 资源限制 `--memory`、`--cpus`、`--pids-limit` 防炸机(fork bomb、死循环编译)。
- `--network none` 默认断网(demo 任务不需要外网),同时断掉数据外发和挖矿两条路。
- 容器内用非 root 用户,挂载只给工作区、不给宿主敏感路径。

两种架构要能比较:①agent 进程在宿主机、只有 `run_shell` 通过 `docker exec` 进容器(改造小,fs 工具走共享 volume);②整个 agent 进程都进容器(隔离彻底,但 API key 也进了容器)。本项目取①够用,取舍理由要能说清。

### diff 事件的生成与渲染
最出彩的功能,原理很朴素:`edit_file`/`write_file` 执行前读旧内容、执行后读新内容,用 `difflib.unified_diff` 生成补丁文本,包成 `diff` 事件(带 `file_path`、`old_content`/`new_content` 或 unified 文本)下发。前端用现成 diff 组件渲染 before/after。要点:大文件要截断(只发变更 hunk ± 上下文);二进制文件跳过;一次任务多次改同一文件时,每次 diff 独立展示(执行过程的时间线)而非合并。

### checkpoint → 任务历史(W6 资产变现)
W6 的 checkpoint 把 `messages + state` 序列化到磁盘,当时是为了 `--resume`;现在同一份数据支撑「任务历史列表 + 历史详情回放」——这验证了 W6 自测题「checkpoint 还能支撑哪些产品功能」的答案。实现上只需:
- checkpoint 里补充事件流的持久化(`events.jsonl` 逐条追加,或从 messages 重建事件)。
- 后端加两个只读接口:`GET /api/tasks`(列表:id/描述/终态/时间)、`GET /api/tasks/{id}`(整段事件用于回放)。
- 前端渲染组件按「数据源无关」设计:实时流和历史回放喂的是同一种事件数组,组件一套复用。

### 部署拓扑与 SSE 的代理坑
前端 Vercel(静态 + Edge,免费额度够)、后端 VPS 或 Railway(要跑 Docker,**Serverless 平台跑不了 docker,别把后端也扔 Vercel**)。三个必查项:
- 跨域:前端域名 → 后端 API,FastAPI 配 CORS 中间件,白名单只放你的 Vercel 域名。
- SSE buffering:反向代理(nginx)默认 buffering 会把流憋成一次性返回,需要 `X-Accel-Buffering: no` 响应头或关 `proxy_buffering`。
- 空闲超时:部分平台/代理对长连接有空闲超时,要定期发心跳注释行(`: ping`)保活。

### 框架对照:LangGraph 与 Claude Agent SDK(面试能聊即可)
W12 末花 2–3 天速览,不深学。LangGraph:把 workflow 显式建模成图(节点 = 步骤,边 = 控制流),state 在节点间传递、内建 checkpoint/human-in-the-loop——对照点是你 W5 手写的 evaluator-optimizer(你用 while 循环 + Pydantic state,它用图 + reducer)。Claude Agent SDK:你的 agent-mini 本质是 Claude Code 简化版,SDK 把 loop、工具、权限、compact、subagent 全部产品化——对照自己讲「SDK 替我做了什么、我手写时怎么做的」(loop=W3、compact=W7、subagent=W6、权限=W10),这半天在面试里性价比极高。

## 4. 自测题 + 答案要点

**1. 为什么 agent 执行环境必须容器隔离?不隔离会怎样?**
- Web 版意味着不受信任的输入:陌生人下发任务 → agent 执行任意 shell → 等于把服务器 shell 公开给全网(读密钥、挖矿、跳板攻击)。
- W10 的白/黑名单是策略层,可被模型生成的花式命令绕过;容器是机制层强制隔离,策略 + 机制才是纵深防御。
- 隔离还要配资源限制(memory/cpu/pids)和默认断网,否则「隔离但炸机」或「隔离但外发数据」。
- 反面推论:纯本地 CLI 自用时可以不隔离(W3–W10 就没隔离),信任边界变了,安全要求才变——能讲清这个变化比背结论重要。

**2. 前端如何区分并渲染 text / tool_call / diff 这些不同类型的流式事件?**
- 后端先定死结构化事件 schema(每条带 type/task_id/seq),SSE 下发;前端解析后按 `type` 分发:text 追加进流式文本块、tool_call 生成 pending 卡片、tool_result 按 tool_use_id 回填对应卡片、diff 渲染 before/after、done 结束 loading 并显示终态。
- 状态管理用 reducer:事件是唯一事实源,UI 全部由事件数组派生——这也天然支持历史回放(同一组件换个数据源)。
- 保序靠 seq,结束靠显式 done 事件而非断流;Vercel AI SDK 的 data stream protocol 是同思路的标准化实现,自定义事件走 data parts。

**3.(补充)为什么要先冻结事件 schema 再开工,而不是前后端边写边改?**
- 事件 schema 是前后端唯一契约,边写边改会导致双端反复返工,两周排期扛不住;这是主文档明确的常见坑。
- 冻结后双端可并行:后端对着 schema 发,前端可先用手造的假事件流开发组件。
- 变更走版本号,历史 checkpoint 里的旧事件才能继续回放(数据兼容性)。

**4.(补充)done 事件为什么必须在异常路径也保证发出?**
- 前端唯一的终态依据是 done;不发,页面永远 loading,用户无法区分「还在跑」和「后端崩了」。
- 实现上放 `try/finally`:正常结束发 success,异常发 failed(带错误摘要),max_turns 发对应状态。
- 同理 checkpoint 的 events.jsonl 也要落终态,否则历史列表里出现「僵尸任务」。

**5.(补充)docker exec 模式下,fs 工具为什么可以留在宿主机执行?这样安全吗?**
- 工作区是共享 volume:宿主机路径和容器内 `/workspace` 是同一份文件,fs 工具在宿主侧读写等价于容器内读写。
- 安全前提是 W10 的 path sandbox 仍然生效:fs 工具被限制在 workspace root 内,越界路径直接拒绝——策略层管 fs,机制层管 shell,各司其职。
- 真正的任意代码执行只发生在 `run_shell`,而它已被 docker exec 圈进容器;若未来给 fs 工具加执行语义(如写入可执行 hook),就要重新评估这个边界。

## 6. 常见坑速查

- **功能贪多做不完**。怎么发现:第 11 周过半时 Session 4(容器)还没动工,或 backlog 里出现三个功能之外的条目(多用户、登录、暗色模式…)。怎么解决:严格砍回三个功能;diff 视图最出彩,资源不够时优先保它,历史列表最后做(它复用现成组件,成本最低)。
- **没做容器隔离就上线**。怎么发现:上线前自问「陌生人提交 `cat ~/.ssh/id_rsa` 会发生什么」,并真的测一次恶意任务。怎么解决:Session 4 是前端开工的前置门槛,绝对红线;部署平台跑不了 Docker 就换平台,不反过来砍隔离。
- **SSE 多事件类型没设计好协议 → 前端渲染混乱**。怎么发现:前端代码里出现对文本内容做字符串嗅探来猜类型,或双端为字段名反复对齐。怎么解决:回到 Session 1,schema 冻结 + 版本化,所有渲染分支只 switch `type` 字段。
- **反向代理 buffering 把 SSE 憋成一次性返回**。怎么发现:本地流式正常,部署后事件全在任务结束时一起蹦出(W1 老坑的部署版)。怎么解决:nginx 加 `X-Accel-Buffering: no`、关 `proxy_buffering`;加心跳注释行防空闲超时。
- **公开 URL 被滥用刷爆 API 费用**。怎么发现:Langfuse 里出现非 demo 的高频任务,或日成本告警。怎么解决:预置 demo 免输入、自由输入加速率限制、复用 W4 单任务成本熔断、给 API key 设月度预算上限。
- **前端让 AI 写完不 review,事件消费逻辑有隐性 bug**(乱序渲染、result 找不到 call)。怎么发现:手造乱序/缺失事件的假流灌给 hook,看 UI 是否崩坏。怎么解决:UI 组件可以放手,`useTaskStream` 这类协议消费核心自己过一遍并配假流测试。
